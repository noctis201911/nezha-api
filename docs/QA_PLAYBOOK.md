# 哪吒全面 QA Playbook（主动发现版）

> **这份文档解决一个具体教训**：定位授权 bug 通过了商家端端到端、后台 admin、顾客端深度 QA 全部验收，却没人发现——因为那些 QA 都在问"功能对不对、走不走得通"，而这个 bug 的答案是"对、走得通，但每次回首页都烦人地重弹"。它是**状态型 ∩ 平台型**缺陷，对快照式 QA 三重隐身。
>
> **配套关系**：
> - 本篇（QA_PLAYBOOK）= **主动出击**：把整页/整流程当真人用户挖一遍，找"能用但烂/烦/丑/反直觉"的问题。
> - `docs/WRAPUP_CHECKLIST.md` = **收尾自查**：汇报"做好了"前过一遍自己刚改的活。
> - `CLAUDE.md` 哪吒 H5 产品标准 = **验收红线**：本篇的"排版/交互轴"就是把那份标准变成可逐条执行的检查。
>
> **维护约定**：每发现一类新的失败模式（像这次的"会话型权限重弹"），就回来加一条轴或检查项 + 记第 8 节 CHANGELOG。这份会越用越准。

## 0. 一句话入口指令（复制即用）

**给用户**：你不用懂下面所有细节。要 QA 时，把下面这句话发给我，按需替换 `<目标>` 即可。

### 全量版（深度 QA，单页/单流程，约 15-40 分钟）
```
对 <目标页面/流程，如「首页」「加购到下单结算全流程」「商家端接单确认收款」> 做哪吒全面 QA：
严格按 docs/QA_PLAYBOOK.md 第 2 节的 15 条失败模式轴逐条走，重点别漏「重复/返回/会话轴(C)」和「真机/平台轴(E)」；验"成功态"必从**标准路由首帧**冷进（不用直链/预热态抄近路，C 轴）；
每条断言都要给证据（截图 / console 输出 / 计数 / 复现步骤），不接受「我觉得没问题」；
发现问题按第 5 节分级：🔴当场修+真机复验，🟡🟢列清单给我拍板；
最后输出第 4 节规定的「分轴报告」，并把我必须亲手补验的项（第 6 节）单独列给我。
```

### 快速版（改完一个小功能后顺手扫一遍，约 5 分钟）
```
对刚改的 <功能> 做哪吒快速 QA：至少走 A快照 / C重复返回 / D状态切换 / J造假翻译 四条轴，每条给证据，发现即修+复验。
```

> 入口指令本身不是魔法——它管用是因为背后挂着这份结构化 playbook。**没有任何一句话能让 AI 凭空像真人一样发现所有问题（原因见第 7 节诚实交底）**，但这套"逐轴 + 要证据"的套路，本来就会逮到这次的定位 bug。

---

## 1. 核心原则（为什么这样组织）

1. **按"失败模式"组织，不按"区域"组织。** 你之前按商家端/后台/顾客端分工 QA，把每个屏都验过，照样漏掉每一个时间型/平台型 bug——因为它们不属于任何一个"区域"，属于"重复使用"和"特定设备"这些横切维度。本篇按 bug **怎么才能被发现**来分轴。
2. **"能用" ≠ "做完"。** HTTP 200、流程走通、build 成功，都只证明"没崩"，不证明"好用、不烦、不丑、不反直觉"。验收标准是**真人会不会皱眉**。
3. **证据铁律：每条断言配证据。** 截图 / console 原文 / 计数（点击数、弹窗数、请求数）/ 可复现步骤。"我觉得 OK" = 没验。这条和 CLAUDE.md「改好了=Playwright截图+console无报错」一脉相承。
4. **发现即分级，不混为一谈。** 🔴必修 / 🟡建议 / 🟢吹毛求疵，分开报，避免用 nitpick 淹没真问题，也避免把真问题混在建议里被忽略。
5. **不造误报。** 不确定的标 🟡 不当 🔴；不无中生有凑数；真没问题就说"这条干净"。

## 2. 失败模式轴（主体 · 15 条）

> 每条给：**为什么**（漏了会怎样）+ **查什么**（检查项）+ **怎么观察**（工具/手段）+ **要什么证据**。
> 标 ⭐ 的是"快照式 QA 最容易漏、本项目已踩过坑"的轴——任何 QA 都不许跳过这几条。

### A. 快照轴（静态单屏）
- **为什么**：排版/视觉问题一张图就看见，是基本盘。
- **查什么**：横向溢出、文字截断/重叠、图裂或占位绿块、白屏/404/500、英文残留/乱码、重复控件（重复标题/客服入口/购物车）、对齐错乱、安全区被状态栏/刘海盖住。
- **怎么观察**：`verify2.js` 移动+桌面截图；DOM 扫 `document.body.scrollWidth > clientWidth`（横向溢出）；扫重复文案。
- **证据**：移动(390×844)+桌面截图各一张；溢出/重复的 DOM 查询输出。

### B. 单流程轴
- **为什么**：按钮没反应、走到死胡同、流程断裂，是"功能根本没通"。
- **查什么**：每个关键按钮点了有反应；流程能从头走到尾不卡死；返回/关闭都有出路（无死胡同）；提交后有明确反馈（成功/失败/loading）。
- **怎么观察**：Playwright 真实点击走一遍；留意点击后 200ms 内有无视觉反馈。
- **证据**：流程关键步骤截图序列 + 每步 console 无 error。

### ⭐ C. 重复/返回/会话轴 ←（这条就是这次定位 bug 的盲区）
- **为什么**：绝大多数"状态型"bug 只在**第二次**才暴露：重复访问、返回上一页、刷新、切后台再回、会话/权限过期后重来。第一次通常是干净的——**但"看似第一次"若是走直链/预热态抄的近路，标准路由的真·首帧可能恰恰是坏的**（见下"标准路由首帧"项）。
- **查什么**：
  - 同一操作**做两遍**结果是否一致（不该重复弹窗/重复请求/重复 toast/重复红点）。
  - **离开再返回**该页：状态是否还在？是否重新弹了不该弹的东西（授权、引导遮罩、登录框）？滚动位置/已填表单是否丢失？
  - **流程中途刷新**：购物车/已选地址/表单内容是否还在？
  - **返回去向＝来处(return-to-origin)**：从不同来处 X 深入功能页 Y（订单/首页/我的 → 客服·地址·优惠券·退款），点关闭/返回键/iOS左滑后是否回到**各自来处 X**——而非固定页/消息列表/白屏。重点盯**自管 popstate/history 的页**（聊天会话、自接历史栈的弹层）与**深链直达**（referrer 空）。先摊「来处×功能页×退出动作」矩阵、**登录态**逐格真点（禁游客代理），记“落到哪 vs 期望来处”。已知坑：客服直达 /chat?direct=1 返回落消息列表（Chat.js 未消费 direct，2026-06-25 修 5babbc5）。
  - **切到后台几分钟再切回**（移动端高频）：页面是否重新挂载、是否重新索取权限、是否还能继续操作。
  - **权限/登录态过期后**：是否优雅提示，还是默默失败或反复弹窗。
  - **⭐ 标准路由首帧（入口保真·冷首帧）**：验"成功态"（页面/功能已渲染出正确成功结果）这类断言时，**必须从真实标准导航入口冷进去看第一帧**——如「订单列表 → 点某单进详情」，**禁止用直链直达 / 注入 localStorage 态 / 已预热缓存抄近路代替**。直链和预热态常"碰巧是干净的"（数据已缓存、loading 守卫已填好、redux 已有值）；而标准路由首帧带的是列表传来的半份 props / 冷 redux / 未 hydrate 的态——坏往往就坏在那一帧。**抄近路验过 ≠ 成功态成立**：成功态类验收至少各留一格"标准入口 + 冷首帧"真走一遍。
- **怎么观察**：Playwright 里 `page.goto` 同页两次 / `page.reload()` / 切 context；**真机上手动离开 App 隔几分钟再回来**。拦截 `getCurrentPosition`/关键接口计数，看第二次是否多调。
- **证据**：两次访问的计数对比（如本次 `geoCalls` 第一次 vs 第二次）；返回后截图；丢状态的复现步骤。

### ⭐ D. 状态切换轴
- **为什么**：同一个页在不同状态下是不同的页。只验一种状态 = 漏掉其余所有状态的 bug。
- **查什么**（CLAUDE.md「三态分开验」的展开）：
  - 登录态：登出 / 已登录 / 登录态过期。
  - 数据态：空（无餐厅/空购物车/无订单/无消息）/ 假数据态 / 真实数据态 / 超长内容（超长店名/地址/备注会不会撑破）。
  - 权限态：未授权 / 已授权 / 已拒绝 / 撤销后。
  - 网络态：在线 / 慢网 / 离线 / 接口报错。
  - 区域态：在配送区内 / 区外（"该地区暂不可用"是否正确触发）。
- **查什么·空态专项**：每个空态必须有**中文占位**，不留空白（例「商家出餐后这里会更新取餐号」）。
- **怎么观察**：注入不同 localStorage（`zoneid`/`location`/登录 token）；Playwright `context.setOffline(true)`；mock 接口返回空/报错。
- **证据**：每种状态各一张截图；空态占位文案截图。

### ⭐ E. 真机/平台轴 ←（iOS 这一整类坑只在这暴露）
- **为什么**：无头/桌面浏览器自动处理权限、不模拟 iOS 会话权限、不重现键盘/安全区/CDN 缓存。本项目已踩过：iOS 定位会话重弹、iOS 键盘顶起穿帮、CF 边缘缓存吐旧图/旧 404、nginx 缓存 404 陷阱。
- **查什么**：
  - **真 iPhone Safari** 上亲自走一遍（不只无头）：定位/相机/通知权限的授予与**过期重问**；键盘弹起时页面是否露馅/背后穿帮/横向溢出；刘海与底部安全区是否处理。
  - **真 Android Chrome**：权限模型与 iOS 不同（一次授予永久），对比行为差异。
  - **多尺寸**：小屏(360)/大屏/横屏；折叠是否错乱。
  - **CDN/缓存**：换 build 后旧资源 404、覆盖同名图后吐旧图——直连 :3000 vs 经 CF 对比（套路见 memory「CF缓存」系列）。
- **怎么观察**：`verify2.js` 用 iPhone UA + 390×844 模拟是**起点不是终点**；关键权限/键盘类问题**必须真机**（这部分要用户亲手做，见第 6 节）。
- **证据**：真机截图/录屏；CF BYPASS vs HIT 对比 curl。

### F. 烦人度/交互摩擦轴
- **为什么**：这次定位 bug 本质是"烦"。技术上能用，体验上每次都戳你一下。这类只有"数一数"才暴露。
- **查什么**：完成核心操作要几步（登录/加购/下单/取餐应 **≤2 步**）；弹了几次窗/授权/确认（多余的确认就是摩擦）；整页刷新了几次（`window.location.reload` 滥用会闪屏丢状态）；重复要求用户输入已经给过的信息；关键按钮（加购/去结算/登录/取餐）是否曾被隐藏或需要滚动才找到。
- **怎么观察**：走流程时计数：点击数、弹窗数、reload 数、必填字段数。
- **证据**：计数表（操作 → 步数/弹窗数），对照"应 ≤N"。

### G. 排版/UI 规范轴
- **为什么**：把 CLAUDE.md 产品标准从"原则"变成"逐条可查"。
- **查什么**（命中即不合格）：品牌红只用于 CTA/徽标/关键状态，优先 `#C4193E`，避免大块 `#E53935` 铺满；背景浅灰 `#F7F8FA`、卡片白底、轻提示暖黄 `#FFF7E6`；圆角 ≥12px；主标题左对齐加粗；关闭 X 在左上角；字体 PingFang SC；移动端弹层走全屏或底部抽屉（不要"内容往下撑开"）。
- **怎么观察**：截图肉眼对照效果图；DOM 取 `getComputedStyle` 抽查关键元素的圆角/背景/对齐。
- **证据**：截图 + 关键元素的 computed style 抽样。

### H. 交互设计轴
- **为什么**：流程"能走通"不等于"顺手"。这条查的是路径设计本身。
- **查什么**：重要操作 ≤2 步；关键按钮永不隐藏；空状态必有中文占位且给"下一步"指引；弹层走全屏/底部抽屉不撑开；收纳成常显行+抽屉的组件，显示条件要带上同款内部 flag（否则 flag 关=空抽屉死胡同，见 memory「收纳行要镜像组件内部门槛」）；返回/关闭/取消都有明确去处；破坏性操作有二次确认+可撤销。
- **怎么观察**：走流程 + 主动尝试"误操作""中途反悔""点错再退回"。
- **证据**：异常路径的截图 + 是否有出路的说明。

### I. 功能设计轴（不只是"实现了没"，是"设计得对不对"）
- **为什么**：最容易被"我把需求做出来了"掩盖。功能存在 ≠ 功能合理。
- **查什么**：边界值（数量 0/最大、金额为负/超大、超长文本、特殊字符、emoji）；并发与重复提交（连点下单会不会重复下单/重复扣减）；幂等（刷新支付页会不会重复触发）；误操作可逆（删地址/取消订单有无确认+能否撤销）；默认值是否安全（开关默认开还是关，默认开会不会误伤真实顾客）；时区/货币/小数（֏ 显示 NaN 类，见 memory「结算页无头验证」）；权限越界（A 用户能不能看到 B 的订单/地址）。
- **怎么观察**：故意输入极端值；连点；并发两个标签页；改 URL 里的 id 试越权。
- **证据**：每个边界的输入→实际行为记录；越权尝试的接口返回。

### ⭐ J. 内容/翻译/造假轴
- **为什么**：本项目硬红线。出现即算"没做完"。
- **查什么**：英文未翻译/文案乱码（注意 `t()` 查不到 key 会回退显示 key 本身且 capitalize，像"没翻译"，见 memory「i18n 在 zh-CN.js」）；**造假**——假红点/假 99+/假"永远有活动"/假未读数（红点通知只反映真实后端状态）；**重复**——重复标题/重复客服入口/重复购物车/残留旧控件；登录注册出现手机号/OTP 或任何非"邮箱+Google+Apple"的入口（红线）。
- **怎么观察**：扫页面所有可见文本挑非中文/乱码；检查红点/角标数据来源是真接口还是写死。
- **证据**：可疑文案截图 + 红点数据来源（接口 or 硬编码）的代码定位。

### K. 性能/加载轴
- **为什么**：慢和闪烁也是体验缺陷，真实流量下还可能拖垮 SSR（本项目已因 SSR 自调用打满 php-fpm 致首页空）。
- **查什么**：首屏可见时间；有无骨架屏/loading（还是白屏干等）；大图是否压缩/懒加载；慢网下是否可用；列表滚动是否卡顿；SSR 是否重复回调自身接口（memory「减SSR自调用」）。
- **怎么观察**：Playwright 限速（slow 3G）跑首屏；看 Network 瀑布；`BUILD_ID` 自检。
- **证据**：慢网截图 + 首屏时间；Network 关键请求耗时。

### L. 可访问性/可读性轴
- **为什么**：移动端手指点不准、对比度低、字太小，都是真实顾客的痛。
- **查什么**：可点击元素点击区 ≥44×44px；文字与背景对比度足够（浅灰字配白底常太淡）；最小字号可读（正文别 <12px）；图标有文字辅助（纯图标按钮老人看不懂）；输入框有 label/placeholder。
- **怎么观察**：DOM 量关键按钮尺寸；抽查对比度。
- **证据**：过小点击区/过淡文字的元素清单 + 实测值。

### M. 安全/数据/合规触点轴
- **为什么**：L1 红线，碰了要先停下报批（见 INVARIANTS.md）。QA 时只做"触点排查"，不改机制。
- **查什么**：页面/接口是否泄露 PII（手机号、地址、邮箱明文出现在不该出现处）；前端是否硬编码密钥/token；登录入口是否只有"邮箱+Google+Apple"（无手机/OTP）；资金相关页是否符合"平台不碰钱、退款原路、USDT 只退原地址"的展示；隐藏 navbar 的页是否自拉 config（否则社交登录段消失=踩红线，见 memory「隐藏navbar的页必须自拉config」）。
- **怎么观察**：扫前端源码找明文密钥；看登录页入口；查 redux global 是否为 null。
- **证据**：发现的泄露/红线点 + 代码定位；**命中 L1 立即停下报用户，不自行改**。

### N. 错误/异常态轴
- **为什么**：顺风顺水都对，一遇异常就白屏/卡死的页太多。
- **查什么**：接口失败/超时/4xx/5xx 时页面是否优雅降级（有重试/有提示，不是白屏或永久 loading）；断网时；空数据时；超长文本/超大数字；表单非法输入（空、超长、特殊字符、SQL/XSS 串）是否被挡且有友好提示。
- **怎么观察**：mock 接口返回错误/超时/空；`context.setOffline`；表单灌脏数据。
- **证据**：每种异常的页面表现截图。

### ⭐ O. 跨界面一致性/不变量轴 ←（商家新单横幅"幽灵数/死胡同" bug 的盲区）
- **为什么**：同一个事实（计数 / 红点 / 未读数 / 余额 / 状态）常在多个界面或组件各算各的；只要两处口径不一致，就出现"这里说有 N 个、点进去却没有"——报假数 + 走死胡同（犯产品红线）。单屏 QA 永远发现不了，因为每个屏单看都"对"。
- **查什么**：① 列出所有显示"同一个数"的地方（侧栏徽标 / 红点 / 横幅 / 列表头计数 / 统计卡 / 角标），确认它们**同源同口径**取数；口径不同必须能解释差异。② 点"红点/计数"跳过去的列表，**单数必须 == 那个数**（点进去空 = 死胡同，必修）。③ 代码味道：grep 面向用户的**裸 `DB::table(...)->count()` / 手写聚合**——它绕过了模型全局作用域(ZoneScope)/业务作用域(NotDigitalOrder / Notpos 等)，与走 Eloquent 模型的列表口径必然分叉。
- **怎么观察**：造**"中间态/脏数据"**让一条记录"被一个口径计入、被另一个口径隐藏"——离线待付款单 / POS 单 / 跨 zone 单 / 待审核 UGC / 软删记录，再比对每个显示该数的地方是否还对得上。
- **证据**：同一时刻各处计数的截图/数值并列 + "点计数→落地列表单数"对照。

## 4. AI 执行规程（收到入口指令后必须这么做）

1. **确认范围**：复述要 QA 的页面/流程 + 从第 3 节矩阵取出该流程的必走轴。范围太大就提议拆分。
2. **逐轴执行，边走边留证据**：每条轴跑对应检查项，证据（截图/console/计数）存到临时目录。**无头模拟是起点，C/E 轴里只有真机能验的项，明确标"待用户真机补验"**（第 6 节），不假装验过。
3. **每发现一个问题，立刻分级**（第 5 节）并记录：现象 + 复现步骤 + 影响 + 证据 + 根因（先取地面真相再下结论，见 memory「调试先取地面真相」）。
4. **🔴 当场修 + 真机复验**（遵守构建排队/精确 commit/合规报批等铁律）；**🟡🟢 列清单给用户拍板**，不擅自改设计取向。
5. **输出"分轴报告"**，固定格式：

   ```
   # 哪吒 QA 报告 — <目标> — <日期>
   范围：<页面/流程> | 必走轴：A B C D E ...
   
   | 轴 | 检查项 | 结果 | 证据 | 处理 |
   |----|--------|------|------|------|
   | C  | 返回首页是否重弹授权 | ❌ 重弹 | geoCalls 0→1, 截图 | 🔴已修(commit xxx)+复验PASS |
   | A  | 横向溢出 | ✅ 干净 | scrollWidth==clientWidth | - |
   ...
   
   🔴必修(已处理)：...
   🟡建议(待拍板)：...
   🟢吹毛求疵：...
   ⚠️ 待你真机补验：...（第 6 节）
   ```
6. **收尾走 `docs/WRAPUP_CHECKLIST.md`**，把临时脚本删干净、残留测试数据清掉、多窗口改动协调好，再汇报。

## 5. 问题分级与处理

- **🔴 必修**：红线（造假/英文残留/白屏/死胡同/登录非邮箱Google Apple/合规）、明确功能缺陷、明确体验伤害（如反复弹授权）。→ 当场修 + 真机复验 + 提交。合规类(M轴/L1)**先停下报批不自行改**。
- **🟡 建议**：体验可更好但不算坏（多一步、提示可更清、排版可更紧）、不确定是否问题的疑似项。→ 列清单 + 给方案，**用户拍板**。
- **🟢 吹毛求疵**：极细微的像素/措辞偏好。→ 一句话带过，不展开、不强推。
- **防误报**：拿不准的标 🟡 不标 🔴；不无中生有凑数；干净就明说"这条干净"。

## 6. 诚实局限 + 你（用户）必须亲手补验的清单

**AI 做不到的，必须老实说，不假装。**

- **我不"过日子"，我"做体检"。** 真人靠长期使用的烦躁感发现"能用但烦"的 bug；我靠某个时刻的检查。**长期烦人度类 bug 我仍可能漏**——这是结构性的，再全的指令也补不满。
- **真机权限/键盘/安全区/推送，无头浏览器测不准。** 这几类**只有你亲自在真 iPhone / 真 Android 上点一遍**才算数。

**所以每次深度 QA 后，我会给你一份"亲手补验"清单，通常包含**（你花 5-10 分钟）：
1. 真 iPhone Safari：定位/相机/通知**授权一次→关 App 隔几分钟/隔一天再开**，看是否还顺、是否重弹。
2. 键盘弹起：在每个输入框（搜索/地址/备注/登录）打字，看页面是否露馅、背后穿帮、横向溢出。
3. 推送：真机收不收得到、点开跳得对不对。
4. 真实下单一笔小额，走完支付→商家确认收款→取餐全链路（资金流只有真跑才算数）。
5. 换 build 后：硬刷新一次，看有没有旧资源 404 / 旧图（CF 缓存类）。

## 7. 为什么没有"一句话就全自动全面 QA"（给新手的诚实交底）

你是新手、只能靠 AI，所以更要把话说透，免得你以为存在一个还没找到的魔法咒语：

- **瓶颈不在话术，在观察手段。** 我只能发现我能观察到的。只要还在桌面无头跑，再华丽的指令也看不见 iOS 会话 bug。要补的是**给我对的设备和场景**（所以有第 6 节），不是更狠的形容词。
- **"找出所有问题"是无界的。** 越要"全部都查"，我越容易要么用 nitpick 淹没你、要么退回泛泛而谈。**这份 playbook 的价值正是把"全面"切成 14 条有界、可执行、要证据的轴**——焦点 + 广度，而不是一句空泛的"全面检查"。
- **它不是 100%，但它点亮了你漏的那一格。** 本次定位 bug，C 轴（做两遍/返回）+ E 轴（真 iOS）任何一条都会逮到它。以后按这份走，同类 bug 跑不掉。
