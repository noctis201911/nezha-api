# 哪吒外卖 — AML / 制裁 人工审核标准作业程序 (SOP)

**版本**: 0.1（初稿） 　**起草日期**: 2026-06-17 　**状态**: 待平台负责人 + 持牌律师/合规顾问复核 　**适用主体**: 哪吒外卖平台风控审核员（管理员）

> 本 SOP 是对外政策 `AML-policy.md` §3 / §7「交易监控与可疑交易处理」的**操作落地**：把政策里的「转人工审核」拆成审核员可以逐步照做、可留痕、可审计的动作。
> 配套文件：`AML-policy.md`（对外政策） · `INVARIANTS.md`（机制变更等级） · `ADMIN_GUIDE.md §4 风控中心`（后台点哪里） · `data-protection.md`（数据保护）。

## 1. 目的

1. 对被自动风控①、退款护栏②、制裁筛查（L1-6）标记为「人工审核 / 暂挂」的交易，提供**统一、可留痕、可审计**的人工处置流程；
2. 确保命中制裁名单或可疑特征的交易，在放行前经过**人工核实**，**合规红线（L1）不被人工误放**；
3. 形成完整审计轨迹，供监管机构、银行、支付伙伴、审计师事后复核。

---

## 2. 适用范围

覆盖**三类**进入人工处置的场景（详见 §4）：下单风控①命中、退款超限②、制裁筛查未决（L1-6 inconclusive）。
适用人员：平台风控审核员（目前 = 平台管理员本人）。

## 3. 角色与职责

> 现实：平台目前由**单人**（负责人本人）操作。本节先列出**理想的职责分离**，标注当前由谁承担，便于将来扩编时落位。

| 角色 | 职责 | 当前承担 |
|---|---|---|
| **审核员（Reviewer）** | 受理队列、核查、初步判定与处置、逐笔留痕 | 负责人本人 |
| **复核员（Checker）** | 对高风险/大额/制裁相关决定二次复核（四眼原则 maker-checker） | 【规划中】暂缺，单人时由审核员自检 + 事后抽查替代 |
| **反洗钱负责人（MLRO）** | 制度维护、上报决定、制裁名单与阈值审定、培训、记录保管、对外联络 | 负责人本人 |

- 单人操作期间的**补偿控制**：① 每笔决定必须写入系统留痕（不靠记忆/口头）；② 制裁与大额决定**当日复盘**；③ 定期（建议每月）按 `docs/QA_PLAYBOOK.md` 抽查既往审核记录。
- **职责分离原则**：发起退款/放行的人与维护制裁名单/阈值的人，扩编后应分离；扩编前以留痕 + 抽查弥补。

## 4. 触发人工的场景 → 入口 → 可用动作（速查表）

| 场景 | 命中规则 (rule) | 后台入口 | 记录状态 | 审核员可用动作 |
|---|---|---|---|---|
| **A. 下单风控①** | `daily_amount`（单日累计超限）/ `frequency`（频次超限）/ `large_amount`（大额特征） | 风控中心 → **审核队列** `/orders/nezha-risk/queue` | `review` / `pending` | **放行** / **退款** / **清退** |
| **B. 退款超限②** | 退款单笔/单日/笔数/窗口超限 | 风控中心 → **退款留痕/审核** `/orders/nezha-refund/records` | `pending_admin` | **放行**（回订单重执行该退款）/ **拒绝** |
| **C. 制裁筛查未决** | `sanction_inconclusive`（USDT 来源地址反查不出，hold 模式暂挂） | 风控中心 → **审核队列** `/orders/nezha-risk/queue` | `pending` | **核实放行+确认收款**（须填核实结论）/ 维持暂挂 |
| **（C-hit）制裁命中** | `sanction`（来源地址命中 OFAC SDN/黑名单） | 自动**拒收**，仅入**风控日志**（不进可放行队列） | `auto`（reject） | 🔴 **不可人工放行**（红线 L1-6） |

> ⚠️ 区分 **C（未决）** 与 **C-hit（命中）**：
> - **命中（`sanction`）**= 系统已自动拒收、订单不放行出餐，**人工无放行入口**，请勿出餐（`ADMIN_GUIDE §4.7.3`）。
> - **未决（`sanction_inconclusive`）**= 来源查不出（无交易哈希 / 链上服务暂不可达），默认 `hold` 暂挂进队列等人工核实（`ADMIN_GUIDE §4.7.4/4.7.5`）。

## 5. 通用审核流程（五步，每场景通用）

```
① 受理  →  ② 核查  →  ③ 判定  →  ④ 处置  →  ⑤ 留痕
```

1. **受理**：进入对应队列，按"先到 / 制裁相关优先"取一笔。记下订单号、账号、命中规则、金额、币种、时间。
2. **核查**：依场景做最小必要核查（§6 分场景）。涉顾客联系的，**只用平台客服会话**，**不得在询问中暗示其正被 AML/制裁审查**（防泄密，见 §8）。
3. **判定**：对照 §7 决策矩阵给出结论：放行 / 退款（原路）/ 清退 / 拒绝 / 升级上报。**存疑从严**（fail-closed）：拿不准就不放行、升级。
4. **处置**：在后台点对应按钮执行。**红线动作（放行制裁相关、退款）须再次自检 §11**。
5. **留痕**：确认系统已记录处理人、动作、理由/核实结论、时间（制裁相关**必填核实结论**）。系统外不得"私下放行"。

## 6. 分场景处置 SOP

### 6.A 下单风控①命中（审核队列）

适用：`daily_amount` / `frequency` / `large_amount`，状态 `review/pending`，**顾客在放行前拿不到收款码**。

1. 打开 **风控中心 → 审核队列**，看该行命中规则、金额、账号、关联订单。
2. **核查要点**（判断是否真实餐饮消费 vs 资金搬运）：
   - 该账号近期订单是否**金额异常、与外卖客单价不符**、或**多笔小额拼凑规避单笔限额**（结构化分拆 smurfing 迹象）；
   - 收货地址 / 商家 / 时间是否合理，是否同一人高频下单到同一收款方；
   - 必要时经**平台客服会话**核实（话术中性，见 §8）。
3. **判定与处置**（§7 矩阵）：
   - 正常消费被误拦 → **放行**（本次豁免，宽限期内同用户不再审核）；
   - 明显异常但未达拒绝 → 暂不放行，**升级**复盘 / 视情 §8；
   - 已收款但应退 → **退款**（务必原路，见 6.B 红线）；
   - 恶意/滥用账号 → **清退**。
4. 留痕：动作 + 理由写入记录。

### 6.B 退款超限审核②（退款留痕/审核）

适用：退款金额/笔数/窗口超限，状态 `pending_admin`。本场景**核心是守 L1-2/L1-3 退款红线**。

1. 打开 **风控中心 → 退款留痕/审核**，定位 `pending_admin` 行。
2. **强制核对三条退款红线**（任一不满足即 **拒绝**，不得放行）：
   - **原路**：法币退还原付款人；**USDT 退回由原始付款交易链上反查出的 from 地址**（系统已锁定，后台无自由填地址入口）；🔴 **禁止退到任何第三方账户/钱包**；
   - **金额 ≤ 原订单**；
   - 退款是否真实合理（订单状态、是否重复退）。
3. 处置：
   - 合规且确需放行 → 点 **放行** → 系统**回订单详情页重新执行该退款**（限额已对该订单豁免）；
   - 不合规 / 存疑 → 点 **拒绝**，并经客服向顾客说明走人工申诉。
4. 留痕：退款留痕表只保存处理退款争议所必要的字段；原始/退款 tx、锁定地址、截图与自由文本在目的用尽或顾客注销后清除，金额、订单、商家、时间、状态等必要事实去主体化保留。只有实际报告主体或其它明确法定义务成立时才适用固定五年下限。

### 6.C 制裁筛查未决 `sanction_inconclusive`（审核队列）🔴 最高优先

适用：USDT「确认收款」时**来源地址反查不出**、`hold` 模式暂挂。**这是制裁合规的咽喉**，按"查不出即不放行"从严。

1. **铁律先行**：🔴 **绝不能因为"反正也查不出来源"就放行**。查不出 = 默认不放行。
2. 打开 **风控中心 → 审核队列**，命中规则显示 `sanction_inconclusive`，取来源信息（交易哈希 / 链 / 订单号）。
3. **核实来源地址不在制裁名单**：
   - 在 **风控日志 → 制裁命中** 比对，或对照 **OFAC SDN 名单**（数字货币地址条目）核实该 from 地址**不在**名单；
   - 若当时是**链上服务不可达**导致反查失败 → 等链上恢复后**重新点「确认收款」让系统自动再筛一次**，能筛出来就走正常自动流程，不必人工放行。
4. **处置**：
   - 已核实来源**不在**名单 → 在该行**填写核实结论（必填，作为合规留痕）** → 点 **「核实放行+确认收款」**。
     - 🔒 系统会**重新链上筛查一次**：此刻反查若**命中制裁名单则仍自动拒收**（人工无法放行已命中的单），否则放行并确认收款。
   - 无法核实 / 存疑 → **维持暂挂**，**请勿出餐**，升级 §8。
5. 留痕：核实结论 + 处理人 + 时间写入 `nezha_risk_records`，长期保存。

## 7. 决策矩阵

| 情形 | 处置 |
|---|---|
| 真实餐饮消费、被规则误拦 | **放行**（下单①）/ 正常确认（制裁未决经核实无命中） |
| 退款合规（原路 + ≤原单 + 真实） | **放行**（退款②，回订单重执行） |
| 退款触红线（非原路 / 退第三方 / 超原单） | **拒绝**（退款②）🔴 |
| 已命中制裁名单 `sanction` | 系统**自动拒收**，🔴 **不得人工放行** |
| 来源查不出、无法核实 `sanction_inconclusive` | **维持暂挂 / 不放行**（fail-closed） |
| 结构化分拆 / 制裁规避 / 与外卖严重不符的异常 | 不放行 + **升级 §8（可疑交易上报评估）** |
| 恶意滥用账号 | **清退** |
| 拿不准 | **不放行 + 升级**（存疑从严） |

## 8. 可疑交易上报（STR/SAR） 【规划中 · 待法律确认】

> ⚠️ 当前平台仅有**内部**审核与留痕，**尚未**建立向主管机关报送的对外机制。本节描述**目标流程**，落地前须经当地律师确认平台的登记/报送义务。

1. **触发评估的情形**（升级到 MLRO）：明显结构化分拆规避限额、疑似制裁规避、与外卖业务严重不符的大额/异常资金特征、同一收款方异常聚集等。
2. **内部升级路径**：审核员 → MLRO（当前同一人，须**当日**形成书面评估并留痕）。
3. **上报决定**：MLRO 评估是否达到向 **亚美尼亚中央银行 金融监测中心（FMC）** 报送的门槛；门槛、格式、时限 **待法律确认**后写入本节。
4. **防泄密（Tipping-off）** 🔴：
   - **不得**向客户、商家或任何无关方透露其交易正被 AML / 制裁审查或上报；
   - 对客户**统一中性话术**："您的订单正在审核中，请联系平台客服"——**不提"制裁/洗钱/上报"**。
5. 上报与评估记录单独留痕、长期保存、访问受限。

## 9. 留痕与记录要求

- **每个人工决定都必须落系统留痕**，不得仅口头/系统外决定。必含字段：订单号、账号、支付通道、金额、命中规则、**处置动作**、**理由 / 核实结论**、处理人、处理时间。
- 制裁相关决定**必填核实结论**（系统已强制）。
- 留存表与期限（见 `AML-policy.md §5/§8`、`INVARIANTS L1-4`）：
  - 风控命中 `nezha_risk_records`（含制裁命中来源地址/tx/链快照）——长期保存；
  - 退款链上留痕 `nezha_refund_records`——按税务、会计、争议或主管机关要求逐字段定期；不得统一豁免 PII 清除；
  - 制裁名单 `nezha_sanction_addresses`。
- 全部经 git 版本管理，变更可追溯。

## 10. 时限与 SLA（建议值，待负责人拍板）

| 场景 | 建议时限 | 说明 |
|---|---|---|
| 制裁未决 `sanction_inconclusive` | **尽快，原则上当日内** | 顾客付款被卡，但**红线优先**——核实未完成宁可继续暂挂，不得为"快"而放行 |
| 下单风控①审核 | 宽限期内处理（当前**放行宽限期 60 分钟**，后台可调） | 误拦应尽快放行以免误伤真实顾客 |
| 退款超限②审核 | 1 个工作日内 | 顾客退款体验 |

> 以上为建议；正式值由负责人确定后回填，并视 SLA 调整 `INVARIANTS` L2 的宽限期参数。

## 11. 红线与禁止动作（处置前必自检）🔴

1. **绝不人工放行已命中制裁名单（`sanction`）的交易**；制裁未决须核实来源不在名单、系统重筛通过后方可放行。
2. **退款绝不改退第三方、绝不超过原订单、绝不脱离原路**（L1-2 / L1-3）。
3. **不得为"让流程顺畅"而擅自关闭制裁筛查 / 退款护栏 / 风控总开关**；这些是 L1/L2，关闭须先告知用户、留痕、记 `CHANGELOG.md`（关闭制裁筛查 = L1-6 失效，属重大合规风险）。
4. **不得泄密**（§8.4）：不向客户/商家透露其被 AML/制裁审查或上报。
5. **不得在系统外私下放行**：所有放行/退款/拒绝必须经后台按钮执行并留痕。
6. 调整任何阈值/开关 → 按 `INVARIANTS.md` 分级流程（L1 须用户批准）。

## 附录 A：审核员速查卡（一页 · 可打印）

> 本卡为正文摘要，仅供手边速查；**遇分歧以正文 §1–§13 为准**。卡随正文更新（同一文件，改正文时一并改卡）。

### 🧭 五步口诀
**① 受理 → ② 核查 → ③ 判定 → ④ 处置 → ⑤ 留痕**　|　两句铁律：**存疑从严**、**查不出不放行**。

### 📋 三类场景速查

| 场景 | 后台入口 | 状态 | 怎么做（关键动作） |
|---|---|---|---|
| **A 下单风控** | 风控中心 → **审核队列** | `review/pending` | 看金额/频次/账号 → 误拦**放行**·应退**退款(原路)**·滥用**清退** |
| **B 退款超限** | 风控中心 → **退款留痕/审核** | `pending_admin` | 先核三条退款红线 → 合规**放行(回订单重执行)**·不合规**拒绝** |
| **C 制裁未决** | 风控中心 → **审核队列** | `pending`（`sanction_inconclusive`） | 核实来源**不在 OFAC 名单** → **填核实结论(必填)** → 点**「核实放行+确认收款」**(系统重筛,命中仍拒) |
| **C-hit 制裁命中** | 仅 **风控日志** | 自动 reject | 🔴 系统已拒收,**无人工放行入口**,请勿出餐 |

### 🔴 处置前红线自检（三秒过一遍）

1. **绝不**人工放行已命中制裁名单的单；制裁未决须核实+系统重筛通过才放。
2. 退款**只原路、≤原单、不退第三方**——任一不满足即拒绝。
3. **不私下放行、不为顺畅关风控/制裁开关、不泄密**（对客户只说"订单审核中,请联系客服"）。

### ⏱️ 时限（建议·待定稿）

制裁未决=**当日内**（红线优先,核不完宁可继续暂挂）｜下单审核=**宽限期内**（当前 60 分钟）｜退款超限=**1 个工作日内**。

### 🆙 何时升级（停手找 MLRO）

明显分拆规避限额 / 疑似制裁规避 / 与外卖严重不符的大额异常 → 不放行 + 升级评估（外部 STR 上报为【规划中·待法律确认】）。
