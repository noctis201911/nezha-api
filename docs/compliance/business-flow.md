# 哪吒外卖 — 业务与资金流说明 (Business & Fund Flow)

**版本**: 1.3 　**更新**: 2026-07-19

> 供监管/银行/伙伴理解平台如何运作、资金如何流动、平台在其中的角色。核心:**平台不碰资金,只撮合与收佣。**

## 1. 各方角色
| 角色 | 说明 |
|---|---|
| 平台 (哪吒外卖) | 撮合顾客与商家、提供下单/配送/客服、按单收佣。**不归集、不代收代付顾客资金。** |
| 商家 (埃里温餐馆) | 提供餐品、**用本人账户直接收取顾客货款**、按原路规则处理退款。 |
| 顾客 (当地华人) | 下单、向商家本人账户付款(人民币或 USDT)、上传付款凭证。 |

## 2. 资金流 (B 方案 — 直付商家)
```
顾客 ──(货款, 人民币微信/支付宝 或 USDT)──▶ 商家本人账户
                                              (平台不经手)
商家 ──(B2B 佣金, 预存佣金按单扣)──▶ 平台
```
- 顾客货款**直接进商家本人账户**,平台不在中间归集 → 不构成二次清算。
- 平台收入 = 向商家收的佣金(默认 10%),通过商家预存的**预存佣金（履约保证金）**按单自动扣除;余额不足则暂停接新单。
- 商家充值预存佣金 = 合法 B2B 交易,非二清。

> **术语说明**: 本机制是"商家先预存一笔款、平台按单从中扣除应收佣金、余额不足即停止接新单"。它兼具**预付**(预存佣金)与**履约担保**(保证金)双重性质,故面向商家/运营统一称**预存佣金**,本合规文件首次出现处并列标注**(履约保证金)**。最终法律用词以专业律师审定为准。

## 3. 计价与币种
- 菜单以**德拉姆(֏/AMD)为基准**计价,价格旁同时展示¥/$换算(USDT ≈ 美元 1:1)。
- 人民币付款顾客按平台公布的当日汇率,看到"应付人民币"折算。
- 系统为单币种记账,按付款方式(人民币/USDT)区分通道。

## 4. 下单与支付流程
1. 顾客选餐 → 结算页(地址/时间/金额明细) → 选人民币或 USDT → 提交下单;
2. **(风控①)** 系统对下单意图做风险预检:命中拒单/审核则不展示收款码;放行则继续;
3. 顾客看到**该商家本人**的收款信息(人民币收款码 / USDT 地址)。USDT 仅对登录顾客开放：顾客先粘贴/扫码并完整确认“本单退款接收地址”，系统分别签发商家收款地址凭据和顾客退款地址凭据；顾客转账并上传凭证时，两份凭据与 user/order/method/network 在同一数据库事务消费，之后退款目标不可修改；
4. **商家确认收款** → 履约配送;
5. 平台按单从商家预存佣金扣佣(开启后)。

> **第 4 步实现说明(2026-07-23 更新)**:商家在自己账户收到顾客转账后,直接在**商家后台「订单 → 待确认收款」**打开该单。人民币仍由商家核对凭证；USDT v2 在确认收款前必须由服务端再次取得终局链上事件并冻结实际支付的 USDT 原子金额，未知、错地址、错合约或未终局均不能确认。平台管理员后备路径与商家路径共用 `OrderLogic::confirm_offline_payment`，不能绕过该金额快照。**全程平台不碰资金(L1-1)**：平台只读公链并记录证据，钱始终直付商家本人。

## 5. 平台对商家的管理抓手 (不持币仍可治理)
- 预存佣金(欠佣即停接单);
- 流量/订单开关;
- 配送派单。

## 6. 已知合规设计
- 平台不持币 → 规避二清/帮信/外汇风险(详见 `AML-policy.md`);
- 退款仅原路退回(合规红线 `../INVARIANTS.md` L1-2/L1-3);
- USDT 不提供兑人民币/提现到中国的入口(L1-3)。

**USDT 退款 v2（代码可部署、生产默认不接新单）**：

1. 新付款只有在 `nezha_usdt_refund_binding_mode=enforce` 且 `nezha_usdt_refund_legal_gate=approved` 时开放；发布默认 `drain/pending`，律师意见后置期间不接新 USDT。
2. 退款记录按付款时冻结的 USDT 原子金额与可退 AMD 快照执行 `floor(paid_atomic × refund_amd / refundable_amd_snapshot)`，禁止退款时实时汇率和 float。
3. 退款产生后先进入 `awaiting_customer_reconfirm`。顾客用原登录方式完成短时、单次、绑定 user/order/credential/address fingerprint/network/refund amount 的新鲜认证，只读确认不可变目标；失败、超时、重放或地址失效均保持 hold。
4. 进入商家执行前，服务端还会用本地 OFAC 地址库筛查冻结目标；筛查关闭、最近同步非成功或超过 48 小时、查询异常、命中均保持 hold。只有名单结果 cleared，商家才可向该地址发送界面显示的精确 USDT 数量并回填公开链上哈希；网络、USDT 合约、地址、原子金额和终局性全部通过后才置 `merchant_refunded`。`tx.from` 仅用于付款来源审计/制裁筛查，任何模式都不能成为退款目标。
5. 残余控制如异地绑定/快速更换监控尚未建设，当前不能作为缓解引用；出现首个此类告警或路由/金额不一致即切 `drain`。

## 7. 上线节奏
- 一阶段:免佣免押拉单量;
- 二阶段:佣金扣除 + 预存佣金机制启用;
- 三阶段:视规模引入持牌分账/聚合支付正规化(届时请专业律师/会计就主体与跨境合规定方案)。

## 8. 商家安全控制 (2026-07-19 已批准实现，待发布验收)

**收款地址边界**: 商家仍不能任意编辑平台记录的收款地址；现行流程由管理员发起受控变更，商家 owner 只能对待处理申请确认或拒绝。确认和拒绝始终要求 owner 当前密码；仅当 owner 自愿启用了 2FA 时，才额外要求一个新鲜、不可重放的 TOTP 验证码。恢复码不能批准该操作。地址最终生效仍受原有管理员复核和状态机约束，本节不改变平台不碰资金的边界。

**商家 2FA 契约**:
1. `vendors` owner 与每个 `vendor_employees` employee 可在已登录的 Web 个人资料页自愿启用彼此独立的 TOTP secret 和恢复码；未启用者密码通过后正常建立 Web 会话或取得 App bearer token，不受 grace/deadline 遗留字段影响。
2. 已启用者在 Web/App 密码通过后必须完成 TOTP 挑战才建立会话或签发 token。第一阶段不新增 App 内启用或商家自助关闭入口；排期命令永久拒绝写入，任何账号均不得被默认、截止时间或敏感操作间接强制启用。
3. 恢复码只用于登录恢复；并发消费最多成功一次，使用后清除旧 2FA、撤销全部 Web/App 认证并完成当前登录，之后 2FA 保持关闭且重新启用仍属自愿。密码变更/重置、认证器或恢复码更换、账号停用也会撤销对应认证；餐厅停用同时覆盖 owner 和所有 employee。
4. 客服不得索取 TOTP secret、验证码或恢复码。紧急恢复必须由两个不同的 super-admin 批准、填写原因、写审计事件、完整撤销并发送不含秘密的账号通知；不能再以单人 SSH 直接关闭替代。
5. 挑战错误不暴露账号/验证码差异，按账号与 IP 双限流；日志仅记录哈希化的请求来源和白名单元数据，不记录密码、secret、验证码、恢复码、token 或完整 session 标识。

详细状态机、入口矩阵、持久化和发布闸门以 `../MERCHANT_TWO_FACTOR_SECURITY_CONTRACT.md` 为唯一产品/安全契约。当前产品裁决是“不强制，只允许商家自愿启用”；上线状态和遗留排期清理必须以发布证据为准。
