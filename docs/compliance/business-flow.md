# 哪吒外卖 — 业务与资金流说明 (Business & Fund Flow)

**版本**: 1.2 　**更新**: 2026-06-15

> 供监管/银行/伙伴理解平台如何运作、资金如何流动、平台在其中的角色。核心:**平台不碰资金,只撮合与收佣。**

## 1. 各方角色
| 角色 | 说明 |
|---|---|
| 平台 (哪吒外卖) | 撮合顾客与商家、提供下单/配送/客服、按单收佣。**不归集、不代收代付顾客资金。** |
| 商家 (埃里温餐馆) | 提供餐品、**用本人账户直接收取顾客货款**、按原路规则处理退款。 |
| 顾客 (当地华人) | 下单、向商家本人账户付款(人民币或 USDT)、上传付款凭证。 |

## 2. 资金流 (B 方案 — 直付商家)
```
顾客 ──(货款, 人民币微信/支付宝 或 USDT)──▶ 商家本人账户
                                              (平台不经手)
商家 ──(B2B 佣金, 预存佣金按单扣)──▶ 平台
```
- 顾客货款**直接进商家本人账户**,平台不在中间归集 → 不构成二次清算。
- 平台收入 = 向商家收的佣金(默认 10%),通过商家预存的**预存佣金（履约保证金）**按单自动扣除;余额不足则暂停接新单。
- 商家充值预存佣金 = 合法 B2B 交易,非二清。

> **术语说明**: 本机制是"商家先预存一笔款、平台按单从中扣除应收佣金、余额不足即停止接新单"。它兼具**预付**(预存佣金)与**履约担保**(保证金)双重性质,故面向商家/运营统一称**预存佣金**,本合规文件首次出现处并列标注**(履约保证金)**。最终法律用词以专业律师审定为准。

## 3. 计价与币种
- 菜单以**美元为基准**计价(USDT ≈ 美元 1:1)。
- 人民币付款顾客按平台公布的当日汇率,看到"应付人民币"折算。
- 系统为单币种记账,按付款方式(人民币/USDT)区分通道。

## 4. 下单与支付流程
1. 顾客选餐 → 结算页(地址/时间/金额明细) → 选人民币或 USDT → 提交下单;
2. **(风控①)** 系统对下单意图做风险预检:命中拒单/审核则不展示收款码;放行则继续;
3. 顾客看到**该商家本人**的收款信息(人民币收款码 / USDT 地址) → 转账 → 上传凭证;
4. **商家确认收款** → 履约配送;
5. 平台按单从商家预存佣金扣佣(开启后)。

> **第 4 步实现说明(2026-06-16 起)**:商家在自己账户收到顾客转账后,直接在**商家后台「订单 → 待确认收款」**打开该单(可见顾客上传的付款截图),点**「确认收款」**把订单 pending→confirmed、通知顾客「支付已验证」,随后出餐配送;若未收到款/凭证有问题,可**「拒收/打回」**并填原因,顾客据此重传或取消。**平台管理员后台离线核验(`Admin/OrderController::offline_payment`)保留为后备路径**(商家不在线/有争议时由运营代为核验),两条路径共用同一份逻辑(`OrderLogic::confirm_offline_payment` / `deny_offline_payment`),避免漂移。**本步骤全程平台不碰资金(L1-1)**:商家确认的是"我已在自己账户收到顾客货款",钱不经平台、收款人始终=商家本人(收款信息仍由平台后台代录,商家无自助改码入口,见 §8)。

## 5. 平台对商家的管理抓手 (不持币仍可治理)
- 预存佣金(欠佣即停接单);
- 流量/订单开关;
- 配送派单。

## 6. 已知合规设计
- 平台不持币 → 规避二清/帮信/外汇风险(详见 `AML-policy.md`);
- 退款仅原路退回(合规红线 `../INVARIANTS.md` L1-2/L1-3);
- USDT 不提供兑人民币/提现到中国的入口(L1-3)。

## 7. 上线节奏
- 一阶段:免佣免押拉单量;
- 二阶段:佣金扣除 + 预存佣金机制启用;
- 三阶段:视规模引入持牌分账/聚合支付正规化(届时请专业律师/会计就主体与跨境合规定方案)。

## 8. 商家安全控制 (规划, 2026-06-14 定下意图)

**当前状态**: 商家收款信息(收款码/USDT地址/收款人)由**平台管理员在后台代录**,商家端**无自助编辑入口**。
→ 因此"攻击者攻入商家账号篡改收款码截取顾客货款"这一风险**目前已被天然堵住**(商家无修改权限),无需额外审核。

**当开放"商家自助修改收款信息"时(留后期), 同步上线以下"商家安全包":**
1. **改动需后台审核**: 商家提交收款码/USDT地址变更后进入待审核状态, 经平台管理员核准后才生效(防止账号被盗后立即改码截款)。
2. **双向通知**: 变更发起即向 ① 平台管理员 ② 商家登记邮箱 各发一封通知(留痕 + 让商家本人察觉异常改动)。依赖邮件服务(SMTP, 当前 .env 已配置, 启用前需实测发信)。
3. **商家可选两步验证(2FA)**: 把现有管理员 TOTP 2FA 扩展到商家端;**每个商家账号各自 opt-in, 不强制全员**;商家如需要, 平台后台可协助开启, 提升商家安全感。认证器丢失同样可经 SSH 应急关闭。

**说明**: 上述三项是一个整体, 触发点是"商家能自助改收款信息"。在该功能上线前, 它们无实际触发对象, 故不提前单独实现。面向商家的正式操作流程文档待功能落地后另行编写。
