# 哪吒外卖 — 反洗钱与制裁合规政策 (AML / CTF Policy)

**版本**: 1.1 　**更新日期**: 2026-07-23 　**适用主体**: nezha.am 实际运营者（身份待补）

> 本文件描述平台自愿采用的交易风险控制，不等于承认平台属于亚美尼亚 AML 报告主体。现行业务是普通订单撮合且平台不提供支付、转账、托管或加密资产服务；运营者和资金路径变化时须重新判断报告主体资格。

## 1. 平台业务模式概述
哪吒外卖是面向亚美尼亚埃里温华人社群的**外卖撮合平台**,连接当地餐馆(商家)与顾客。平台定位为**纯撮合 + 收佣的市场/SaaS**:
- 顾客付款**直接付进商家本人**的收款账户(人民币:微信/支付宝;USDT:链上地址),**平台全程不归集、不代收代付资金**。
- 平台收入来自向商家收取的佣金,通过商家预存佣金（履约保证金）按单扣除(合法 B2B)。
- 详见 `business-flow.md`。

此模式从结构上**避免了二次清算(无证经营支付结算)**,降低了平台成为资金归集/转移通道的洗钱风险。

## 2. 合规承诺
平台遵守真实适用的反洗钱、反恐怖融资及经济制裁规则，并自愿采用风险监控。只有在报告主体身份、法域连接点和具体义务成立时，才把相应尽调、报告或固定留存期限作为法定义务；不得把条件触发项提前写成永久承诺。

## 3. 交易监控规则 (已实施 — 交易红旗风控①)
平台对每笔下单/支付意图执行自动风险评估,规则阈值集中存于后台配置项、可由管理员调整(非硬编码)。处置分三档:**拒单 / 转人工审核 / 放行**。

| 规则 | 触发条件(默认阈值) | 处置 |
|---|---|---|
| 单笔限额 | 法币 > 等值 100 美元;USDT > 200 美元 | **拒单**,提示联系客服 |
| 单账号单日累计 | 法币 > 300 美元;USDT > 500 美元 | 转**人工审核** |
| 交易频次 | > 5 单/24小时 或 > 2 单/10分钟 | 转**人工审核** |
| 金额特征 | 整百/整千金额、或显著偏离餐饮客单价的大额 | 标记**人工审核** |

- 命中"拒单"或"人工审核"的订单,**顾客在审核通过前无法取得商家收款码,即无法完成付款**("先拦付款"原则)。
- 人工审核由平台管理员在后台「风控审核队列」处理,可**放行 / 退款 / 清退**,每步留痕。
- 放行后该顾客在限定宽限期内可重新完成支付。

> 当前状态:风控引擎与后台已上线,总开关默认关闭(待菜单计价单位标准化后启用)。启用后即按上表执行。

## 4. 制裁与黑名单地址筛查 (现有 OFAC 自愿风控)
针对 USDT 收款通道,平台当前自愿实施:
- 顾客 USDT 付款在「确认收款」环节(放行出餐的闸口),系统反查该笔链上交易的**付款来源地址**(BSC/TRC20,复用退款链上反查设施),比对 **OFAC SDN 制裁名单**(公开数据,含"数字货币地址"条目)及黑名单地址库;
- 地址按格式规范化后比对(EVM 统一小写以吸收 EIP-55 校验大小写差异;Tron base58 区分大小写原样比对);
- **命中即拒收**该笔交易(标记拒收、订单不予确认、不放行出餐)并写入风控记录(命中规则=sanction、来源地址、交易哈希、链、订单号);
- 来源地址无法反查时(无交易哈希 / 链上服务暂不可达)的处置可配置,**默认采取保守的 fail-closed:不放行、转人工复核**(订单暂挂待来源核实,符合"无法筛查即不放行"的制裁合规原则);运营也可按运行实际切换为"放行+留痕待事后核";两种均留痕;
- 名单由后台定时任务每日从 OFAC 公开 SDN 名单刷新。OFAC 只有在美国主体、美国人、美国受控服务商、美元清算、合同承诺或其它美国法连接点出现时才可能成为强制义务；不能只因在亚美尼亚经营或使用 USDT 就称为亚美尼亚法定最低要求；
- 亚美尼亚法要求关注本国及基于联合国安理会决议的指定清单。当前系统尚未把这些清单建立为独立、可验证的数据源和匹配链，正式把筛查称为“满足亚美尼亚法定清单”前必须补齐；
- 总开关默认开启,可在后台风控设置调整;
- 后续可对接专业链上分析服务(Chainalysis / TRM / Elliptic)增强。

## 5. 链上与交易资料留存
- 普通撮合平台并非当然的 AML 报告主体，不能把退款交易哈希、来源/锁定地址、截图、自由文本和全部顾客资料统一保留五年。
- `nezha_refund_records` 与 `nezha_risk_records` 只在税务、会计、退款争议、安全调查或主管机关依法要求所必要的字段和期限内保留；顾客注销后清除主体关联、地址、哈希、截图与自由文本，保留去主体化的金额、商家、时间、状态等必要交易事实。
- 若运营者未来实际成为 AML 报告主体，再按当时适用的客户尽调、交易记录与五年下限调整；不得倒推为当前义务。
- 制裁名单本身(地址、币种、来源、入库时间)留存于 `nezha_sanction_addresses`。

## 6. 退款政策 (合规红线)
- 退款**仅允许原路退回**:原支付方式 / 原卡 / 原钱包 / 原 USDT 付款地址;
- 退款金额**不超过原订单**;
- **禁止退至任何第三方账户/钱包**;
- 因平台不持有资金,退款由商家按上述规则执行,平台系统对退款指令做"原路"锁定与留痕。

## 7. 可疑交易处理流程
1. 自动规则命中 → 写入风控记录(订单号、账号、命中规则、金额、时间、处置结果)+ 推入人工审核队列;
2. 管理员核实(可联系顾客)→ 放行 / 退款(原路) / 清退,操作留痕;
3. 记录只按明确目的和期限保存；不必要的主体、地址、IP、截图和自由文本到期清除。

## 8. 记录与留痕
- 风控命中记录表 `nezha_risk_records`:订单号、账号、支付通道、金额、命中规则(含 sanction 制裁命中)、处置结果、处理人、处理时间;制裁命中含来源地址/交易哈希/链快照。
- 退款与风控记录按 §5 的逐字段目的和期限处理；公开制裁名单本身不含顾客 PII，可按名单同步与审计需要保存。
- 全部通过 git 版本管理,变更可追溯。

## 9. 角色与职责
- **平台管理员**:监控审核队列、处置可疑交易、维护阈值与名单、保管记录。
- **商家**:如实提供本人收款信息,按原路规则执行退款。
