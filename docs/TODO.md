## 1. 商家「确认收款」异常高频告警（风控防滥用）

- **状态**：留后续，**事件触发**（非排期，别拍死日期）。
- **等级**：🟡 L2（风控参数/规则增强）。
- **背景**：2026-06-16「商家自营确认收款」上线（commit `d672602`）。理论风险=商家"未真收到款就点确认"。
- **为什么先不做**：
  1. 虚假确认的经济损失**落在商家自己头上**（要照常发餐），平台/顾客不受损 → 天然自我约束。
  2. B 方案**平台不给商家打款**（二清腿已拔、无 disbursement），刷确认**偷不到平台的钱**；每单还扣商家预存佣金 → 刷单自己倒贴、无动机。
  3. 现网真实 paid 单 ≈ 0，现在建告警 = 在没数据处装一个永不响、还要维护的警报，属过早优化。
  4. 下单前已有**风控①**预检。
- **过渡期不瞎**：确认/拒收已**留痕**（`logs` 表记 who / when / 前后状态 / `by_type=admin|vendor`），出事后可完整还原。原则=**先记录、后告警**。
- **触发条件（满足任一即做）**：
  1. 真实订单/确认量起来，够让告警在真实数据上有意义（如每天数十笔跨商家确认）。【主信号】
  2. **预存佣金扣佣正式开**（`nezha_deposit_mode_status`=1）——确认开始牵动真实佣金记账。【最自然的搭车时点】
  3. 出现"按订单数发的"促销 / 补贴 / 排名——首次给"刷确认"造出收益动机。
  4. 疑似商家账号被盗等安全事件。
- **真做时的范围（很轻，数据已现成）**：每日 / 准实时检查，标记"某商家每小时确认数超阈值"或"下单→确认间隔短得不合理（盖章式确认）"，结果挂到后台**风控中心** + 可选邮件给运营。复用现有风控阈值设置页，**不需新表、不改订单状态机**。
