# 顾客自动注销 V6：亚美尼亚最低合规完整方案

状态：实现中，全部开关默认关闭，未部署。父任务批准：2026-07-23。

## 1. 完成口径

只有同时满足以下条件才算完成：代码与字段矩阵落地；MySQL 5.7 up/down/up、并发、崩溃续跑和通知重试通过；H5 目标手机视口截图闸通过；staging 固定 SHA 全生命周期与 72 小时真实 soak 通过；运营者真实身份完成中、亚文披露；生产固定 SHA 发布后真实页面、API、通知、日志和回滚复验通过。任一项缺失都不得开 production 闸。

## 2. 事实—缺口矩阵

| 事实/要求 | 修正前缺口 | V6 处理 |
|---|---|---|
| 处理目的完成后应终止，非必要/违法数据应更正、封锁或销毁 | 仅清 users、地址和少量订单字段 | 清理支付/退款自由文本、地址与哈希、举报主体、对话、UGC 和附件；只留去主体化最小交易事实 |
| 销毁后应及时告知；项目门采用 3 个工作日 | 没有通知表、发送 actor 或送达证据 | 独立加密 outbox；完成后进入待发，失败退避重试，逾期 critical；发送/撤销后清收件地址 |
| 顾客撤销必须是明示动作 | 后续未勾选订单会静默取消 | 新订单只新增 blocker 并重启完整倒计时；仅登录确认页或“我的”按钮可撤销 |
| 顾客应能知道当前状态 | V5 移除了 Profile 状态 | “我的”展示等待、待开闸、倒计时、法定暂停、执行/清除状态及显式撤销 |
| 72 小时是产品便利，不得吞掉法定请求 | UI/接口未区分 | 隐私政策和 UI 明示正式请求走 support@nezha.am，按法定期限独立处理 |
| 不得产生半注销状态 | execution 开、purge 关会先封号 | 四闸：intake → countdown → execution → purge；不可逆执行同时要求 execution+purge |
| 通知需在销毁后仍可达 | Telegram-only/未验证邮箱账户无可靠通道 | 首期只受理有已验证邮箱的顾客；地址应用层加密，发送或撤销后立即清除 |
| 运营者信息应可识别 | 法定全称、注册号、地址仍未知 | 生产硬停；不得造假或用品牌名长期替代 |

## 3. 产品流程

1. 登录顾客在结算页展开“隐私保护”，只有已验证邮箱时可主动勾选。
2. 成功下单后在同一订单事务内建立/重申请求和加密通知记录；失败下单不产生请求。
3. 订单、退款、余额、申诉、客服或依法冻结任一未结时保持等待。新增事项会清空旧倒计时。
4. blocker 全清且 countdown 闸开启时，撤销全部会话并开始完整 72 小时。登录只展示“继续登录并取消注销 / 暂不登录”，不静默改变请求。
5. 到点后只有 execution 与 purge 两闸同时开启、且 blocker 再次为零，才进入不可撤销阶段。
6. 清除按幂等步骤执行；崩溃后从事件证据续跑。全部完成才标记 completed。
7. outbox 发送中、亚双语完成通知；普通失败自动退避重试，3 个工作日门前必须送达或停产处置。

## 4. 字段留存矩阵

| 数据 | 注销处理 | 保留事实/理由 |
|---|---|---|
| users 姓名、电话、邮箱、头像、第三方身份、推送标识 | 清除并删头像文件，撤销 access/refresh token | 仅保留不可登录的无身份 tombstone ID，供历史外键完整性 |
| 配送地址、坐标、订单备注/配送指示 | 清除 | 无 |
| 订单行 | 去主体化保留 | 金额、商家、时间、状态等税务/会计/争议必要事实 |
| 离线支付与退款 | 清截图文件、付款信息、地址/hash、自由文本、主体关联 | 金额、渠道、状态、订单关联、结构化结果按必要期限 |
| 客服对话与消息 | 删除 | 必要法定冻结时在进入注销前以 blocker 阻断 |
| 评价、本地生活帖子/笔记及附件 | 删除内容与文件 | 无；依法冻结的帖子先阻断 |
| 商家/评价/本地生活举报 | 去举报人主体并清自由文本；被删除内容的从属举报删除 | 结构化理由、对象、状态可在必要期限内保留 |
| 钱包/积分流水 | 余额必须归零后才执行；主体账号已去身份 | 结构化账务事实按实际法定义务 |
| 加密备份 | 14 份滚动窗口；恢复后必须跑恢复重删 | 仅为灾备，不作为绕过删除的历史库 |
| 完成通知收件地址 | 受理时加密；发送或撤销后清除 | 仅为完成通知 |

## 5. 四级发布闸与回滚

- intake：只开放受理，执行链仍关闭。
- countdown：允许 blocker 清零的请求开始完整 72 小时，会话撤销；不开时停在 `ready_for_countdown`。
- execution：表示允许到点 claim，但代码仍要求 purge 同时开启。
- purge：与 execution 同时满足后才允许封号和清除。

回滚优先关 intake/countdown/execution/purge，并切回 previous API/Web release。已经受理的状态和事件表保留，不做破坏性 down；已有真实行时 migration 明确拒绝回退。恢复备份后运行 restore replay，保证已完成顾客不会因恢复重新出现 PII。

## 6. 生产硬停

- 运营者法定全称、注册号、注册/办公地址、有效联系方式未提供，或未进入中、亚文政策；
- 任一字段无法说明处理目的和期限；
- MySQL 5.7 加密表、并发、恢复重删或通知重试失败；
- staging 全链、目标视口截图、console/page/network、72 小时 soak 或回滚不通过；
- 出现资金路径、依赖升级、不可逆迁移或范围外生产数据写入。
