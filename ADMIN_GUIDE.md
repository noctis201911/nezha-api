### 4.6 退款控制（原路锁定 + 限额）— 2026-06-14 新增

> 这是**退款合规红线（L1）的执行机制**。和上面的"下单风控"是**两个独立开关**，互不影响。

**4.6.1 设置在哪里**：侧边栏 → **订单** → **风控中心** → **风控设置** → 页面下方「**退款控制**」卡。

| 设置项 | 默认 | 含义 |
|---|---|---|
| **退款护栏总开关** | **0（关）** | 关=退款行为和以前完全一样；开=启用下列所有限制 |
| 单笔退款上限 (֏) | 40000（≈$100） | 超过转审核（不直接拒） |
| 单商家单日退款累计上限 (֏) | 120000（≈$300） | 超过转审核 |
| 单商家单日退款笔数上限 | 5 | 超过转审核 |
| 退款窗口（交付后 N 天内可退，0=不限） | 7 | 超窗口转审核 |
| USDT 退款链上自动校验 | 1（开） | 校验退款 tx 的金额+地址；关=只锁定+人工核 |
| BscScan / TronGrid API Key | 空 | **留空即可**（用免密钥公共节点）；被限流时再去免费注册填入 |

**开关开启后会强制**：退款金额 ≤ 原订单；USDT 退款目标=由原始付款链上反查的 from 地址（**禁止自由填地址、禁止退第三方**）；法币退还原付款人（政策+凭证）；超限退款转审核队列。**全程留痕（≥5年）**。

**4.6.2 退款留痕 / 审核**：侧边栏 → **风控中心** → **退款留痕/审核**（`/orders/nezha-refund/records`）。

- **看什么**：每笔退款的订单/商家、通道、退款额/原单、**原路锁定地址**、**链上校验状态**、处置状态。
- **USDT 登记退款哈希**：商家原路退款后，把商家给的**退款交易哈希**填进该行的输入框 → 点「链上校验」。系统自动核对：收款地址是否==原付款地址、金额是否≥应退额。
  - 绿色 `verified`=金额+地址都对；红色 `failed`=对不上（**别放行，核对是否退错地址/退第三方**）；黄色 `manual`=链上没查到，需人工核。
- **传凭证**：人民币退款截图 / USDT 辅助截图，可在该行上传。
- **超限审核**：状态为 `pending_admin`（超限）的退款，点「放行」后**回订单详情页重新执行该退款**（限额已对该订单豁免）；或点「拒绝」。

**4.6.3 开启退款护栏的步骤（真实影响操作，先验证）**

1. 先用**一笔测试订单**走通：在测试单上做一次退款，到「退款留痕」确认记录正确、（USDT）链上校验通过；
2. 确认「退款控制」里的限额对你的客单价合理；
3. 把**退款护栏总开关从 0 改为 1**，保存。
4. ⚠️ 开启属"真实影响"——会立即对所有真实退款生效。拿不准就先保持 0（关），不影响现有退款。

## 5. 业务设置

### 5.1 货币（2026-06-14 以德拉姆 ֏ 计价；顾客端将另显示换算 ¥/$）
计价基准货币是 **AMD（亚美尼亚德拉姆，符号 ֏）**：商家按德拉姆录入菜价，商家后台 / 报表金额按 ֏ 显示。
- 之前没设置，系统默认回退成美元 $；现已设为 AMD（期间试过人民币 ¥，最终定为以德拉姆计价）。
- **在哪里改**：后台 业务设置 → 货币 / Currency（对应业务设置项 `currency`，值 = `AMD`）。
- ⚠️ 改币种**只换显示符号、不换金额数字**：商品价格本身不会自动换算。所以菜价请直接按"德拉姆"录入（一份兰州拉面填 `1500` 就是 ֏1500）。
- ✅ **顾客端 H5 已显示换算后的人民币(¥)与美元($)价格**（2026-06-14 上线）：浏览/菜单/购物车显示"≈¥/≈$"参考价；结算页显示**精确应付 ¥ / USDT**（即顾客实付、商家实收的金额）。
- **汇率在哪维护**：后台 → 餐厅 → 任一餐厅 → **收款信息** tab → 「**顾客端换算汇率**」卡，填 `1 元人民币 = ? ֏`、`1 美元 = ? ֏`（平台全局，所有餐厅共用；当前 1元=55֏、1美元=400֏，对应业务设置项 `nezha_rate_cny_to_amd` / `nezha_rate_usd_to_amd`）。⚠️ 平台不经手资金、商家按 ֏ 标价却收 ¥/USDT，**此汇率即事实结算汇率、商家承担汇率波动**，请定期校准别让它过期。
- ✅ **关联提醒（已处理）**：风控阈值原按美元设计，2026-06-14 已改为德拉姆量级（单笔 200000 ֏ / 单日 800000 ֏；退款 40000 / 120000 ֏，见 4.2 / 4.6）。开启风控总开关前仍请按真实客单价复核一遍（见 4.4）。当前两个风控总开关都=0（关），不影响现状。

### 5.3 附加费名称（2026-06-15 改为「活动期免收平台服务费」）
「附加费 / Additional Charge」是可加在顾客订单上的一笔平台服务费，名字由总后台统一设置（业务设置项 `additional_charge_name`），显示在商家「订单报表」列头（及顾客账单对应行）。
- **当前状态**：开关 `additional_charge_status` = 空（**关**）。代码里只有开关 = 1 时才会真扣这笔费用，所以**现在每单附加费恒为 0、顾客一分钱没多收**（拉新期暂不收）。
- 2026-06-15 应「拉新期暂不收服务费」，把名字从英文默认 `Additional Charge` 改为 **「活动期免收平台服务费」**（旧值：`Additional Charge`）。只改显示名字，不动收费逻辑。
- **在哪里改**：后台 业务设置 → 附加费名称字段（对应 `additional_charge_name`）。
- 🔴 **恢复收费时的红线顺序（务必照做，否则等于欺骗顾客）**：等拉新活动结束、要重新收平台服务费时——
  1. **先**把名字从「活动期免收平台服务费」改回中性名（如「平台服务费」）；
  2. **再**把开关 `additional_charge_status` 开成 1 并设好金额。
  顺序反了（名字仍写「免收」却已开始扣钱）= 顾客账单出现「免收平台服务费 ¥X」，属**造假红线**。

### 5.4 商家收款方式配置（支付宝 / 微信 / USDT）〔2026-06-15 微信独立成单独收款码〕
顾客在结算页可选三种线下收款方式，**钱直接付进商家本人账户、平台不碰钱（B 方案）**。每家餐厅的收款码/地址需后台代录。
- **在哪里录**：后台 → 餐厅 → 选某餐厅 → **收款信息** tab。页面分三张卡：
  1. **支付宝收款（人民币）**：填「收款人姓名」+ 上传「支付宝收款码图片」（对应字段 `payee_name` / `rmb_qr_image`）。
  2. **微信收款（人民币）**：上传「微信收款码图片」（对应字段 `wechat_qr_image`）。**与支付宝码分开上传**。
  3. **USDT 收款**：选网络（TRC20/BSC）+ 填收款地址（`usdt_address` / `usdt_network`）。
- 🆕 **2026-06-15 变更**：此前微信/支付宝共用一张「人民币收款码」，现已拆成两张独立码。顾客结算页选「支付宝」显示支付宝码、选「微信」显示微信码。
  - ⚠️ **微信码留空的后果**：顾客在结算页能看到「微信」方式，但选中后会显示「本店暂未设置微信收款码」——**等于这家店收不了微信**。请给每家店都补上微信码，或提醒商家提供本人微信收款二维码（见商家手册）。

## 6. 后台安全

### 6.1 两步验证 (2FA) — 强烈建议开启
给后台登录在密码之外加一道手机认证器动态码，密码即便泄露别人也进不来。

**怎么开启：**
1. 登录后台后，浏览器打开：`https://api.nezha.am/admin/two-factor/setup`
2. 用手机装一个认证器 App（Google Authenticator / 微软 Authenticator / 1Password 均可）。
3. 用 App 扫页面上的二维码（扫不了就手动输下面那串密钥，类型选"基于时间/TOTP"）。
4. 把 App 上显示的 6 位码填进"第二步"输入框，点"确认并启用"。
5. **页面会显示 8 个一次性恢复码——立刻抄下来存好（密码管理器/纸条）。这是认证器丢了时的备用钥匙，关掉页面就再也看不到。**

**以后登录：** 输完邮箱密码后，会多一步要你输认证器上的 6 位码。

**认证器丢了 / 被锁在外面怎么办（三选一）：**
- 用一个**一次性恢复码**代替 6 位码登录（每个只能用一次）。
- 都没有了：让懂技术的人通过服务器 SSH 执行一条命令关闭你的 2FA：
  `cd /www/wwwroot/api.nezha.am && php artisan nezha:2fa-disable 你的后台邮箱`
  执行后即可仅用密码登录，再重新设置 2FA。

**关闭 2FA：** 在同一设置页输入当前登录密码确认即可关闭。

> 说明：2FA 默认是关的，开不开由你决定；不开的话登录跟以前完全一样。

### 6.2 顾客付款凭证自动清除（隐私合规）
顾客上传的付款凭证（转账截图/付款信息）属个人数据，系统会**每天自动清除超过保留期的旧凭证**（只抹凭证内容+截图，保留订单/交易记录供对账），满足"个人数据到期删除"的合规要求。
- **保留天数**：默认 **90 天**。如需调整，改业务设置项 `nezha_payment_proof_retention_days`（数字，单位天）。
- 链上/交易/审计记录按合规另行留存 ≥5 年，不在自动删除范围内。

### 6.3 数据库加密 + 每日加密备份（已自动运行，无需后台操作）
数据库里所有顾客信息（姓名/电话/邮箱/地址/坐标/证件号等）现在在硬盘上是**加密存储**的，并且**每天凌晨自动做一份加密备份**。这两件事都在服务器后台自动跑，**后台界面里没有按钮要点**。

**🔑 你唯一要做的一件事：把"备份钥匙"另存一份到安全的地方。**
- 备份钥匙就是服务器上这个文件：`/root/.nezha/backup.key`
- **这把钥匙丢了，那些加密备份就永远打不开了。** 请让懂技术的人把它的内容复制出来，存到你的密码管理器 / U盘 / 离线安全处各一份。
- （数据库本身的加密钥匙在 `/www/server/mysql-keyring/`，由服务器自动管理，正常不用动；换服务器时才需要它，已有备份。）

**备份在哪、留多久：**
- 位置：`/www/backup/database/nezha-enc/`，文件名形如 `sql_api_nezha_am-日期.sql.gz.enc`
- 频率：每天 04:17 自动生成一份，**保留最近 14 份**，旧的自动删。
- 备份是加密的，所以就算这些备份文件被人拷走，没有上面那把钥匙也打不开。

**怎么用备份恢复数据（出事时给懂技术的人看）：**
- 完整步骤写在服务器文件 `/root/nezha-backup/README-RESTORE.txt` 里。
- 简版：用钥匙解密 → 解压 → 导回数据库：
  `openssl enc -d -aes-256-cbc -pbkdf2 -pass file:/root/.nezha/backup.key -in 备份文件 | gunzip > /root/restore.sql`，确认无误后 `mysql` 导入。

**这套加密防什么、不防什么（实话）：** 防的是**硬盘或备份文件被偷/被拷走**；**防不了**服务器本身被黑（程序或 root 被攻破时数据仍是明文可读）。所以它是合规要求的"静态加密"那一层，不能替代日常的账号、密钥、服务器安全。
- 手动预演（看会删什么、不真删）：`php artisan nezha:purge-payment-proofs --dry-run`

### 8.4 用户自助发帖（UGC）与审核 〔2026-06-15 新增〕
除了你手动录入，顾客也可以在 H5「本地生活」自己发帖。**这个功能默认是关闭的**，需要你手动开。

**① 开关：是否允许顾客发帖**
- 位置：本地生活列表页**最上方**有一块「用户发帖入口」卡片，显示当前状态（**已开放 / 未开放**）和一个开关按钮。
- **默认未开放**：顾客端「本地生活」右下角的「发帖」按钮点了只会提示"即将开放"，不会真打开发帖表单。
- 你觉得准备好了（有人手能审核），点 **开放发帖入口**；要暂停就点 **关闭发帖入口**。
- ⚠️ 这是**真实影响开关**：一开，登录顾客就能提交真实信息（含他们的真实联系方式）。开之前请确认你能及时审核。

**② 顾客发的帖默认进「待审核」，不会自动公开**

### 8.5 发帖护栏：违禁词 / 举报 / 限流 / 免责声明 〔2026-06-15 新增·窗口⑤〕

## 9. 推送通知 (Firebase FCM)

顾客端「出餐提醒」等推送，靠 Google Firebase Cloud Messaging (FCM) 发送。**2026-06-15 已配置并真机验证通过**（手机能收到通知）。下面记录配置位置和**踩过的坑**，以后维护别重蹈覆辙。

### 9.1 后台在哪填
- 路径：后台 →「第三方集成与配置」→「**Firebase 推送设置**」（路由 `business-settings/fcm-index`）
- 那页**只有最上面那个大文本框「Service File Content」要填**——粘贴**服务账号 JSON 全文**。
- ⚠️ 下面那排小框（API密钥 / FCM项目ID / 授权域名 / 应用ID 等）**全部留空**。我们顾客端走前端自己的 `.env`，不读后台这些字段，填了也没用、反而易混。

### 9.2 服务账号 JSON 从哪来
1. [Firebase 控制台](https://console.firebase.google.com) → 项目 **nezha-dummy** → ⚙️ 项目设置 →「服务账号」标签 →「**生成新的私钥**」→ 下载一个 JSON 文件。
2. 用记事本打开，全选复制，粘进 9.1 那个大文本框 →「提交」。
3. 🔴 这个 JSON 含私钥 = **密钥**，只贴这一个地方，**别贴聊天 / 别进 git / 别外发**。

### 9.3 🔴 维护红线（都是 2026-06-15 配置时真踩过的坑）
1. **绝不删除服务账号 `firebase-adminsdk-fbsvc@nezha-dummy...`**。在 Google Cloud「服务账号」页删除时，"删钥匙"和"删账号"挨得很近——删了账号会导致发送报 `account not found`，整个推送瘫痪。只在必要时删**单个钥匙行**，永不删账号。
2. 服务账号必须有 **「Firebase Cloud Messaging API Admin」** 这个 IAM 角色，否则发送报 `403 cloudmessaging.messages.create denied`。若账号被重建过，需在 Google Cloud → IAM 重新授予此角色。
3. 换/转钥匙时：先生成新钥匙→上传后台→**确认能发**，再删旧钥匙。别先删后换。
4. 前端 `.env.production` 里有 6 个 `NEXT_PUBLIC_FIREBASE_*` + `NEXT_PUBLIC_FIREBASE_VAPID_KEY`（gitignored，服务器有 `.bak` 备份）。SW 文件 `public/firebase-messaging-sw.js` 里也硬编码了同一套配置，**两处的 appId 要一致**。

### 9.4 顾客侧前提（顾客自己操作，非后台）
- **iPhone 必须"添加到主屏幕"、从主屏幕图标打开**，才能收推送（iOS 死规矩；普通 Safari 标签页不行，需 iOS 16.4+）。安卓 Chrome 无此限制。
- 顾客需在站内**授权通知权限**，前端才会把设备 token 上报到后端（存 `users.cm_firebase_token`）。

### 9.5 怎么自检推送是否正常
- 商家把某单推进状态（接单/出餐），已授权的顾客手机应收到通知。
- 后端发送逻辑在 `app/CentralLogics/Helpers.php::sendNotificationToHttp`（FCM HTTP v1，OAuth2 服务账号鉴权）。
