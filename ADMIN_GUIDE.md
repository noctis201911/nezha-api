## 3. 订单管理

- 路径：侧边栏 → **订单**
- 可查看所有订单状态、手动确认、退款操作

### 3.1 离线支付核验（现以商家自营为主，后台为后备）〔2026-06-16 起〕

顾客离线支付（支付宝/微信/USDT 转账 + 上传截图）的核验，**现在主要由商家本人在商家后台「订单 → 待确认收款」自行确认**（商家核对自己账户到账后点「确认收款」）。后台这条核验路径**保留为后备**，用于商家不在线、或某笔有争议时由运营代为处理：

- 路径：后台侧边栏 → **订单 → 离线支付核验**（`离线支付待核验/已核验/已拒绝` 列表）。
- 操作：打开订单 → 看顾客付款截图 → 「是，已收到支付」（转 confirmed、通知顾客）或填原因「未收到支付」打回。
- **效果与商家点「确认收款」完全一致**：两条路径共用同一份核验逻辑，确认人（admin / 商家）会记入留痕（`logs` 表）。
- 🔴 **平台不碰钱**：无论谁确认，核验的只是"商家本人账户是否已收到顾客货款"，平台全程不归集、不代收；钱直付商家本人。后台代核验前，最好先与商家确认其账户确已到账。

### 3.2 订单收尾（顾客确认收货 / 超时自动兜底）〔2026-06-17 上线〕

平台不配送（顾客自叫 Yandex / 自取），配送单**没有骑手点"已送达"**，商家也不知道顾客几点收到。所以一单怎么算"完成"由这两条路径收尾：

1. **顾客「确认收货」（主路径）**：商家把单点到「已出餐 / 待取餐」（handover）后，顾客端出现「确认收货」按钮，顾客收到餐后自己点确认 → 订单转「已送达」。
2. **超时自动兜底**：顾客忘了点时，系统每小时巡检，把 handover **超过 N 小时**（默认 24h）仍无人确认的配送/自取单自动判为「已送达」。

**为什么重要（关系到账目）**：订单走到「已送达」是**佣金结算的触发点**——此时系统对直付单从商家「预存佣金」按佣金率扣一次（`commission_deduction`，扣佣开关 `nezha_deposit_mode_status` 开启时生效）；在此之前不结算。所以**配送单不能永远卡在 handover**，否则佣金永不入账、顾客端永远"未送达"——本功能就是堵这个洞（此前 7 家店均为非自营配送，配送单全部卡 handover）。

**恰好扣一次**：顾客确认、商家标记、自动兜底等所有"送达"路径，都以"该单是否已生成交易流水"为唯一闸门，先到先记、其余跳过，**不会重复扣佣**（已用事务回滚仿真验证：deposit_balance 恰好扣 1 笔、L1-1 平台不碰钱未变）。

**两个可调设置**（`business_settings`，目前经数据库 / 告知开发调整，无独立后台页）：
- `nezha_auto_finalize_handover_status`：自动兜底开关，`1`=开（默认）、`0`=只靠顾客手动确认。
- `nezha_auto_finalize_handover_hours`：自动兜底时长，默认 `24`（小时）。

> ⚠️ **真实影响**：自动兜底默认开启、对真实配送单生效。上线时已确认线上无积压（dry-run 实测无单卡 handover 超 24h），不会误判现有订单。如要关闭或改时长，告知开发调 `business_settings` 对应键。

### 4.6.1 直付单取消/退款 → 自动通知商家退款 + 留痕 〔2026-06-17 上线 · F-4〕

当你在「订单管理」**取消**或**退款**一笔**直付订单（offline_payment，顾客直付商家本人账户）**时：
- 系统会**自动**（无视上面的「退款控制」总开关）生成一条 `待商家退款`(pending_merchant_refund) 留痕记录，并**推送通知该商家**去按原路退还顾客。
- 记录含：应退金额（≤原单）、原付款渠道（微信/支付宝/USDT）、USDT 自动反查的**原始付款地址**（退款只能退这个地址）、操作人，留存 ≥5 年。
- **平台全程不碰这笔钱**：钱一直在商家本人账户，平台只做通知与留痕（L1-1）。
- 商家在**商家后台「订单 → 待退款」**退款后点「标记已退款」，记录转 `merchant_refunded`。你可在数据库 `nezha_refund_records` 表核对（USDT 可凭商家回填的退款 tx hash 链上核验）。

> 这条补「资金闭环 QA」的 F-4 缺口：以前后台取消/退款直付单后，平台既不通知商家去退、也没有"商家已退款"的记录。现在两者都有了，且平台依然不经手资金。

### 4.7 制裁名单筛查（OFAC SDN）〔2026-06-16 新增·合规红线 L1-6〕

> 这是**合规红线**：平台不得与受制裁主体交易。顾客用 **USDT** 付款、商家/你在后台「确认收款」时，系统会反查这笔链上转账的**付款来源地址**，比对 **OFAC（美国财政部）SDN 制裁名单** 里的数字货币地址。**命中就自动拒收**——订单不会被确认、不放行出餐，并记一条「制裁命中」到风控日志。

**4.7.1 设置在哪里**：侧边栏 → **订单** → **风控中心** → **风控设置** → 页面「**制裁名单筛查**」卡。
- **总开关 `nezha_sanction_screen_status`**：默认 **1（开）**。关掉=不再筛查 USDT 来源地址（**L1-6 失效，属重大合规风险，关闭前务必想清楚**）。
- **名单来源 URL**：默认 OFAC 官方 SDN.XML 公开端点，一般不用改。
- **来源地址反查不出时**（`nezha_sanction_inconclusive_action`）：默认 **拦截待人工（hold）** = 查不出来源就不放行、转人工复核（更稳健，见 4.7.4）；可改 **放行+留痕（allow）**。
- 卡片底部显示**当前名单地址数**和**上次同步时间/状态**。

**4.7.2 名单怎么更新**：系统每天 **04:30** 自动从 OFAC 公开名单拉取最新地址（无需人工）。如想立即更新，可在服务器运行 `php artisan nezha:sync-sanction-list`。取数或解析失败时**保留上一次的名单不动**（不会因为一次拉取失败就清空、漏筛）。

**4.7.3 命中了怎么看**：侧边栏 → **风控中心** → **风控日志** → 顶部点「**制裁命中**」筛选。每条记录里能看到命中的来源地址、交易哈希、链、订单号。命中单已被系统自动拒收，**请勿出餐**；如确认是误判需放行，请先核实来源后再人工处理。

**4.7.4 查不出来源地址时**（无交易哈希 / 链上服务暂时连不上）：行为由 4.7.1 的「来源地址反查不出时」开关决定，两种都会在风控日志记一条 `sanction_inconclusive` 待人工复核：
- **拦截待人工（hold，默认）**：**不放行出餐**、中止本次确认。订单保持 `pending`（不是拒收，**来源核实或链上恢复后可重新点「确认收款」再筛一次**）。更符合"查不出即不放行"的制裁筛查通用准则。
- **放行 + 留痕（allow）**：照常确认出餐，只记一条待人工事后核。适合你还没配 TronGrid key、Tron 公共端点常限流、又不想误挂合法 USDT 单的场景。
> 提示：**TRC20（波场）** 来源反查走 TronGrid，未配 key 时公共端点易限流→更常"查不出"；填 **TronGrid API Key**（后台「退款控制」那个框，与本筛查共用）即可大幅缓解。**BSC（币安链）** 来源走公共节点 `bsc-dataseed`（免密钥、稳定），一般不会"查不出"；"退款控制"里的 **BscScan Key 当前不影响本筛查**（代码未用它），可不填。若开 hold 后仍频繁误挂，可临时切 allow。

**4.7.5 未决单一键处理（审核队列）〔2026-06-17〕**：开 hold 时被暂挂的单会进 **风控中心 → 审核队列**，命中规则显示 `sanction_inconclusive`。核实步骤：
1. 拿到该笔 USDT 的**交易哈希**（顾客付款凭证里 / 问顾客），到区块浏览器（BSC: bscscan.com / TRC20: tronscan.org）查这笔转账的**付款来源地址**；
2. 在 **风控日志 → 制裁命中** 或对照 OFAC 名单确认该来源地址**不在**制裁名单；
3. 回审核队列，在该行**填写核实结论**（必填，作为合规留痕），点「**核实放行+确认收款**」——系统会**重新链上筛查一次**：若此时反查出命中制裁名单**仍自动拒收**（人工无法放行已命中的单），否则放行并确认收款。
> 即：这个一键按钮只能放行"查不出"的单，**永远不能放行"已命中"的单**；放行操作记 `reviewed_by`/`reviewed_at`/核实备注，长期留痕。

> ⚠️ **真实影响提示**：本功能默认开启，且已载入 OFAC 现行名单，即刻对真实 USDT 确认收款生效。正常顾客几乎不可能在制裁名单上，误伤风险极低；如需暂停，按 4.7.1 把开关置 0。

### 5.7 顾客端餐厅列表的四种排序（「综合排序 / 距离 / 销量 / 好评率」含义）〔2026-06-16 新增〕
顾客在首页「附近推荐」上方那行排序，决定餐厅列表先后。四种排法下**营业中的店永远排在关店的店前面**，然后：
- **综合排序（默认）**：哪吒的"推荐"排法，对标各大外卖平台的综合排序——按一个「综合分」从高到低。综合分 = **距离(越近越高) 0.45 + 评分(越高越高) 0.30 + 销量(越多越高) 0.25** 三项加权。所以又近、又好评、又有销量的店靠前；很近但评分差的店不会只因为"近"就霸榜。
- **距离**：按离顾客（按其手机定位；未授权定位时按埃里温市中心）从近到远。
- **销量**：按历史总订单量从多到少。
- **好评率**：按顾客评价平均星级从高到低（与餐厅卡片上的星级同口径，无评价的店沉底）。
- **想调综合排序的权重**（比如让"销量"更重要）：三个权重写在后端代码的综合分公式里（不是后台开关），要调跟开发说一声、改一行即可。当前商家普遍还没订单，综合排序主要由"距离+评分"驱动；等真实订单累积后"销量"才会明显发挥作用。

## 6. 后台安全

### 6.1 两步验证 (2FA) — 强烈建议开启
给后台登录在密码之外加一道手机认证器动态码，密码即便泄露别人也进不来。

**怎么开启：**
1. 登录后台后，浏览器打开：`https://api.nezha.am/admin/two-factor/setup`
2. 用手机装一个认证器 App（Google Authenticator / 微软 Authenticator / 1Password 均可）。
3. 用 App 扫页面上的二维码（扫不了就手动输下面那串密钥，类型选"基于时间/TOTP"）。
4. 把 App 上显示的 6 位码填进"第二步"输入框，点"确认并启用"。
5. **页面会显示 8 个一次性恢复码——立刻抄下来存好（密码管理器/纸条）。这是认证器丢了时的备用钥匙，关掉页面就再也看不到。**

**以后登录：** 输完邮箱密码后，会多一步要你输认证器上的 6 位码。

**认证器丢了 / 被锁在外面怎么办（三选一）：**
- 用一个**一次性恢复码**代替 6 位码登录（每个只能用一次）。
- 都没有了：让懂技术的人通过服务器 SSH 执行一条命令关闭你的 2FA：
  `cd /www/wwwroot/api.nezha.am && php artisan nezha:2fa-disable 你的后台邮箱`
  执行后即可仅用密码登录，再重新设置 2FA。

**关闭 2FA：** 在同一设置页输入当前登录密码确认即可关闭。

> 说明：2FA 默认是关的，开不开由你决定；不开的话登录跟以前完全一样。

### 6.2 顾客付款凭证自动清除（隐私合规）

### 6.3 数据库加密 + 每日加密备份（已自动运行，无需后台操作）
数据库里所有顾客信息（姓名/电话/邮箱/地址/坐标/证件号等）现在在硬盘上是**加密存储**的，并且**每天凌晨自动做一份加密备份**。这两件事都在服务器后台自动跑，**后台界面里没有按钮要点**。

**🔑 你唯一要做的一件事：把"备份钥匙"另存一份到安全的地方。**
- 备份钥匙就是服务器上这个文件：`/root/.nezha/backup.key`
- **这把钥匙丢了，那些加密备份就永远打不开了。** 请让懂技术的人把它的内容复制出来，存到你的密码管理器 / U盘 / 离线安全处各一份。
- （数据库本身的加密钥匙在 `/www/server/mysql-keyring/`，由服务器自动管理，正常不用动；换服务器时才需要它，已有备份。）
- 〔2026-06-17 容灾QA〕已把 `backup.key` 与 `keyring.master.backup` 两把钥匙打包下载到站长本机一份；请再各复制一份到密码管理器 / U盘，别和服务器放在一起。

**备份在哪、留多久：**
- 位置：`/www/backup/database/nezha-enc/`，文件名形如 `sql_api_nezha_am-日期.sql.gz.enc`
- 频率：每天 04:17 自动生成一份，**保留最近 14 份**，旧的自动删。
- 备份是加密的，所以就算这些备份文件被人拷走，没有上面那把钥匙也打不开。
- 〔2026-06-17 容灾QA 新增〕现在每天还多备份一份 `nezha-files-日期.tar.gz.enc`：里面是**上传的图片、支付凭证、和 `.env` 配置**（这些以前没进备份，丢了就没了）。
- ⚠️ **当前所有备份都在本机这一台：能防误删，防不住整台机器/硬盘坏掉或被勒索。** 真正的"异地容灾"（把备份推到云存储或另一台机）还没做，等有真实订单量后建议补上。

**怎么用备份恢复数据（出事时给懂技术的人看）：**
- 完整步骤写在服务器文件 `/root/nezha-backup/README-RESTORE.txt` 里。
- 简版：用钥匙解密 → 解压 → 导回数据库：
  `openssl enc -d -aes-256-cbc -pbkdf2 -pass file:/root/.nezha/backup.key -in 备份文件 | gunzip > /root/restore.sql`，确认无误后 `mysql` 导入。

**这套加密防什么、不防什么（实话）：** 防的是**硬盘或备份文件被偷/被拷走**；**防不了**服务器本身被黑（程序或 root 被攻破时数据仍是明文可读）。所以它是合规要求的"静态加密"那一层，不能替代日常的账号、密钥、服务器安全。
- 手动预演（看会删什么、不真删）：`php artisan nezha:purge-payment-proofs --dry-run`

### 8.4 用户自助发帖（UGC）与审核 〔2026-06-15 新增〕
除了你手动录入，顾客也可以在 H5「本地生活」自己发帖。**这个功能默认是关闭的**，需要你手动开。

**① 开关：是否允许顾客发帖**
- 位置：本地生活列表页**最上方**有一块「用户发帖入口」卡片，显示当前状态（**已开放 / 未开放**）和一个开关按钮。
- **默认未开放**：顾客端「本地生活」右下角的「发帖」按钮点了只会提示"即将开放"，不会真打开发帖表单。
- 你觉得准备好了（有人手能审核），点 **开放发帖入口**；要暂停就点 **关闭发帖入口**。
- ⚠️ 这是**真实影响开关**：一开，登录顾客就能提交真实信息（含他们的真实联系方式）。开之前请确认你能及时审核。

**② 顾客发的帖默认进「待审核」，不会自动公开**
- 顾客提交后，帖子状态是 **待审核**（黄），顾客端列表里**看不到**，只有他自己在「我的发布」能看到"审核中"。
- 你在列表用状态筛选点 **待审核**，逐条处理：
  - **通过**（绿按钮）→ 帖子变 **已发布**，立刻出现在顾客端本地生活列表。
  - **驳回**（黄按钮）→ 展开一个输入框填**驳回理由**（可留空，默认套用通用理由）→ 确认 → 帖子变 **已驳回**；发帖用户在「我的发布」能看到"已驳回"和你写的理由。
- 平台帖（你录的）不受审核流程影响，仍用「发布⇄转草稿」切换。

**③ 顾客发帖里的图片和联系方式**
- 顾客可上传最多 6 张图片（自动转 webp 存储），列表卡封面优先用第一张图。
- 顾客填的**联系方式仍是 PII**：列表永不显示、详情只给登录顾客；30 天后由每日任务 `nezha:purge-locallife-pii` 自动清除联系方式+删图（保留帖子行供审计）。详见 data-protection.md。
- 反垃圾：每个顾客**每天发帖上限**（默认 5 条，超出当天拒绝）。更多护栏（违禁词过滤 / 举报 / 限流 / 免责声明）见 **8.5**。

### 8.6 AI 预审工作法（你 + Claude 一起审，省人工）〔2026-06-16 新增〕

你想"让 AI 帮你看帖、看不准的才人工"。这件事**不用改任何代码、不用加状态**——现成的「待审核」状态就够用，做法是约定一套流程：

**怎么用（你开了 UGC 开关、有顾客发帖之后）**
1. 你在 Claude Code 窗口里说一句：**"审一下待审帖"**。
2. Claude 读出所有「待审核」（status=3）的帖子，**逐条**给你：内容摘要 + 判定（🟢正常 / 🔴违禁 / 🟡拿不准）+ 理由。
3. 处置规则（已和你约定）：
   - 🟢 **明显正常** → Claude 直接改成「已发布」，立即对顾客可见。
   - 🔴 **明显违禁** → Claude 直接改成「已驳回」，驳回理由写明 **"【AI预审】…"** 方便区分。
   - 🟡 **拿不准** → **留在「待审核」不动**，等你人工看。
4. 于是你**只需要在后台筛「待审核」这一档**——里面剩下的，天然就只有 AI 拿不准的帖子。你点通过/驳回即可。

**为什么不另加一个"待人工复核"状态**：「待审核」本来就不向顾客展示，AI 处理过的帖子都已离开这档，留下的自然只剩拿不准的，效果完全一样，还少一个状态要维护、少一处可能漏判（前端某查询忘了排除新状态就会把违禁帖漏给顾客）。

**留痕**：AI 驳回时理由带 "【AI预审】" 前缀；通过的帖 Claude 会在审核日志里记一行（帖ID/时间/判定/理由），出问题能查"这条是 AI 放的还是你放的"。

**⚠️ 重要边界（合规）**
- AI 只做**预审分级**，**终审与平台责任仍在你**。监管追责追的是平台主体，不是 AI。AI 会被绕过、会误判，所以 🟡 一律退给人工、不替你冒险。
- 这套**走你本人在 Claude Code 的交互额度**，不是服务器后台 7×24 自动调用。后台全自动值守需要单独接 Claude API（按量付费），量大了再考虑，现在没必要。
- 现在 `locallife_ugc_enabled` **仍是关闭的**（律师审校前不开），待审队列是空的，这套流程是"备好等开闸"的状态。

## 9. 推送通知 (Firebase FCM)

顾客端「出餐提醒」等推送，靠 Google Firebase Cloud Messaging (FCM) 发送。**2026-06-15 已配置并真机验证通过**（手机能收到通知）。下面记录配置位置和**踩过的坑**，以后维护别重蹈覆辙。

### 9.1 后台在哪填
- 路径：后台 →「第三方集成与配置」→「**Firebase 推送设置**」（路由 `business-settings/fcm-index`）
- 那页**只有最上面那个大文本框「Service File Content」要填**——粘贴**服务账号 JSON 全文**。
- ⚠️ 下面那排小框（API密钥 / FCM项目ID / 授权域名 / 应用ID 等）**全部留空**。我们顾客端走前端自己的 `.env`，不读后台这些字段，填了也没用、反而易混。

### 9.2 服务账号 JSON 从哪来
1. [Firebase 控制台](https://console.firebase.google.com) → 项目 **nezha-dummy** → ⚙️ 项目设置 →「服务账号」标签 →「**生成新的私钥**」→ 下载一个 JSON 文件。
2. 用记事本打开，全选复制，粘进 9.1 那个大文本框 →「提交」。
3. 🔴 这个 JSON 含私钥 = **密钥**，只贴这一个地方，**别贴聊天 / 别进 git / 别外发**。

### 9.3 🔴 维护红线（都是 2026-06-15 配置时真踩过的坑）
1. **绝不删除服务账号 `firebase-adminsdk-fbsvc@nezha-dummy...`**。在 Google Cloud「服务账号」页删除时，"删钥匙"和"删账号"挨得很近——删了账号会导致发送报 `account not found`，整个推送瘫痪。只在必要时删**单个钥匙行**，永不删账号。
2. 服务账号必须有 **「Firebase Cloud Messaging API Admin」** 这个 IAM 角色，否则发送报 `403 cloudmessaging.messages.create denied`。若账号被重建过，需在 Google Cloud → IAM 重新授予此角色。
3. 换/转钥匙时：先生成新钥匙→上传后台→**确认能发**，再删旧钥匙。别先删后换。
4. 前端 `.env.production` 里有 6 个 `NEXT_PUBLIC_FIREBASE_*` + `NEXT_PUBLIC_FIREBASE_VAPID_KEY`（gitignored，服务器有 `.bak` 备份）。SW 文件 `public/firebase-messaging-sw.js` 里也硬编码了同一套配置，**两处的 appId 要一致**。

### 9.4 顾客侧前提（顾客自己操作，非后台）
- **iPhone 必须"添加到主屏幕"、从主屏幕图标打开**，才能收推送（iOS 死规矩；普通 Safari 标签页不行，需 iOS 16.4+）。安卓 Chrome 无此限制。
- 顾客需在站内**授权通知权限**，前端才会把设备 token 上报到后端（存 `users.cm_firebase_token`）。

### 9.5 怎么自检推送是否正常
- 商家把某单推进状态（接单/出餐），已授权的顾客手机应收到通知。
- 后端发送逻辑在 `app/CentralLogics/Helpers.php::sendNotificationToHttp`（FCM HTTP v1，OAuth2 服务账号鉴权）。

### 9.6 推送文案（notification_messages 表 / 2026-06-17 修复并补齐）

顾客端每次订单状态变化（下单 / 接单 / 备餐 / **出餐** / 完成 / 取消 / 离线付款确认或拒收 / 退款）要推给顾客的**文字内容**，存在数据库 `notification_messages` 表（`user_type='user'`），多语言译文放 `translations` 表。**这跟 9.1～9.5 的 FCM「发送通道」是两回事**：通道是「怎么发」，本表是「发什么字」。两者缺一不可。

- 🔴 **此表为空 = 所有状态推送静默失效**（2026-06-17 修复）：表曾被导库 / 迁移洗空过一次，导致 `Helpers::order_status_update_message()` 取不到文案返回 null，`sentUserNotification()` 里的 `if ($value && $user_fcm)` 不成立 → **FCM 推送和站内信都不发**。商家点「出餐」顾客端零提醒，根因就在这里（不是通道坏了）。
- **怎么补 / 改文案（两条路）**：
  - ① **后台页**（推荐用于日常改文案）：后台 →「通知」→「Push Notification Messages」（路由 `notification-messages`），按 key 填，含语言 tab，保存即 `firstOrNew` 落库。
  - ② **一键补默认值**（表被洗空时重建，**推荐用于灾后恢复**）：`php artisan db:seed --class=NezhaUserNotificationMessageSeeder`（幂等，可反复跑，不会重复插）。seeder 在 `database/seeders/`，含全部订单状态的中英文默认文案。
- 🔴 **locale 关键坑（务必知道）**：推送查找文案用的是**顾客的 `current_language_key`**，本项目真实顾客是 `en` 与 `zh-CN`（**不是**后台系统语言码 `zh`）。所以：基础 `message` 列存**中文兜底**（覆盖 zh-CN 顾客及任何没对上的语言），译文按 `en` / `zh-CN` 存。**后台语言 tab 只有 `zh`，单靠它填的译文 zh-CN 顾客匹配不到**，会落回中文兜底列（结果仍是中文，但要明白这层机制——别奇怪「我在 zh tab 改了，zh-CN 顾客怎么没变」）。要改 zh-CN 顾客看到的中文，请改**基础/default 那一栏**（= `message` 列）或重跑 seeder。
- **出餐（handover）文案约定**：推送里**不塞取餐号**——取餐号顾客在 App「消息中心」查（出餐后 track 接口才返回，见 commit bdc1b02）。推送只负责「**提醒去看**」，文案如「您在「{restaurantName}」的订单已出餐，请在 App 查看取餐号并尽快前往取餐。」
- **2026-06-17 已补的 key**（`user_type='user'`，`status=1`，各含 `zh` / `zh-CN` / `en`）：`order_pending_message`、`order_confirmation_msg`、`order_processing_message`、`order_handover_message`（出餐）、`out_for_delivery_message`、`order_delivered_message`、`delivery_boy_delivered_message`、`delivery_boy_assign_message`、`order_cancled_message`、`order_refunded_message`、`refund_request_canceled`、`offline_order_accept_message`、`offline_order_deny_message`。
- **文案可用占位符**（`Helpers::text_variable_data_format`）：`{restaurantName}`、`{orderId}`、`{userName}`、`{otp}`（= 取餐号）、`{tokenNumber}`、`{tableNumber}`。⚠️ 值为空时不替换、会原样显示，别用没传值的占位符。

### 9.7 商家端推送文案与送达现状（2026-06-17）

商家端通知文案同样存 `notification_messages`（`user_type='restaurant'`）。2026-06-17 已补 3 条会用到的 key：`restaurant_order_notification`（新订单）、`restaurant_account_block` / `restaurant_account_unblock`（平台暂停 / 恢复店铺）。一键重建：`php artisan db:seed --class=NezhaRestaurantNotificationMessageSeeder`。

- **故意没补**的 restaurant key：提现批准 / 驳回（`restaurant_withdraw_*`—— B 方案平台不碰钱、无提现，填了会与合规模型冲突、误导商家）、活动 / 广告 / 订阅类（MVP 未启用）。
- vendor 语言是 `zh`（注意：不是顾客的 `zh-CN`），文案基础列存中文兜底 + `zh` / `en` 译文。

🔴 **重要：补了文案 ≠ 商家就能收到新订单推送。** 新订单的 FCM 发送在 `Helpers::sentRestaurantNotification()`，当前 B 方案下三条送达渠道都还没通：

| 渠道 | 现状 | 要通需要 |
|---|---|---|
| FCM 设备推送 | ❌ 所有 vendor `firebase_token` 为空 | 商家在网页端授权浏览器通知后、前端把 token 上报到后端 |

### 9.8 启用商家 Telegram 接单提醒（操作步骤）

让某家店在手机上即时收到新订单（最稳的商家提醒渠道，独立于网页 / App 推送）：
1. 让商家用手机 Telegram 搜索并打开机器人 **&#64;Nz_order_bot**，点「开始 / Start」，随便发一句（如「你好」）。
2. 后台进该餐厅：左侧「餐厅」→ 打开餐厅 →「**设置**」标签 → 找到「**Telegram 接单提醒**」卡片。
3. 点「**查看最近联系机器人的会话**」→ 列出刚给机器人发过消息的会话及其 chat id → 点对应那条的「**填入**」（或手动把 chat id 粘进输入框）。
4. 点「保存」。完成。之后该店每来一笔新订单（pending / 确认），机器人私聊推一张卡片：单号 / 类型 / 合计 / 商品 / 时间。
- 留空 = 不发；每笔订单只推一次（已去重）。
- 群发：把机器人拉进店员群、填该群的 chat id（负数）即可，全店都能收到。
- 目前仅平台运营在后台设置；商家端自助绑定后续可做。
- 实现：`Helpers::sendTelegramOrderAlert()`（发送）+ 后台「设置」页卡片（`admin.restaurant.update-telegram` / `admin.restaurant.telegram-recent-chats`）+ 字段 `restaurants.telegram_chat_id`。

## 10. 佣金充值管理（商家预存佣金）

**入口**：后台左侧栏「交易管理」区 →「佣金充值管理」（`/admin/nezha-deposit`）。

**这是什么**：B 方案下平台不碰顾客的钱，平台收入靠向商家收佣金。商家先把佣金「预存」给平台，平台按每单自动扣佣；余额不足会暂停该商家接单。本页用于查看每个商家的预存佣金余额、累计充值/扣佣/退还，并在商家把钱打给平台后**记录充值**。

**一览页能看到**：
- 顶部 4 个汇总卡：全平台预存佣金总额、欠款商家数（余额为负）、累计充值、累计扣佣。
- 每个商家一行：当前余额（**负数红色显示「欠款」**=该商家欠平台佣金）、累计充值、累计扣佣、累计退还、上次充值时间。

**怎么记录一笔充值**（商家把预存佣金转给平台后）：
1. 在该商家行点「记录充值」。
2. 填「充值金额」（= 商家实际打给平台的金额）、可填备注（如「微信转账 6/16 / 凭证号」）。
3. 点「确认入账」→ 商家余额立即增加并写入流水（留痕操作人）。

**全部流水**：点右上「全部流水」或某商家行的「流水」，可按商家 / 类型（充值 / 扣佣 / 退款返还）筛选。

**说明**：
- 充值是合法 B2B（平台向商家收佣的预付），不涉顾客资金、不构成二清（合规 L1-1/L1-5）。
- 扣佣只在业务设置「预存佣金扣佣开关」(nezha_deposit_mode_status) 打开后才发生；当前默认关闭（免佣免押拉量阶段）。
- 余额可为负（= 商家欠平台佣金，已确认采用此策略）；负余额触发停止接单，商家充值后恢复。
